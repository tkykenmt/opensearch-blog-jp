## OpenSearch とは

OpenSearch は、オープンソースの分散型検索・分析エンジンです。2021 年に Elasticsearch からフォークされ、Apache Lucene ライブラリをベースにしています。このプロジェクトは Linux Foundation 傘下の OpenSearch Software Foundation がホストしています。

130 以上の GitHub リポジトリを持つ OpenSearch には、Java、Python、Kotlin、TypeScript、JavaScript など様々なプログラミング言語で構築されたコンポーネントやプラグインがあります。この多様性により、フロントエンド開発者、機械学習 (ML) エンジニア、バックエンドシステムに興味がある方など、誰にでも活躍の場があります。

## なぜ OpenSearch にコントリビューションすべきか

### 1. 学びと成長

コントリビューションを通じて、実際のプロジェクトでの実践的な経験を得られます。ダミープロジェクトとは異なり、コミュニティが直面している実際の問題を解決することになります。

### 2. 好きな技術に取り組める

本業では好みの技術スタックを使えないこともあるでしょう。OpenSearch のようなオープンソースプロジェクトは、情熱を持っている分野を探求し構築するための場となります。私の場合、AI/ML への強い関心が OpenSearch ML Commons プラグインへのコントリビューションの動機となりました。

### 3. 評価を築く

大小を問わず、すべてのコントリビューションが公開ポートフォリオに追加されます。キャリアの可能性を広げ、オープンソースコミュニティでの存在感を確立できます。

### 4. インパクトを与える

あなたのコード、レビュー、ドキュメントは、世界中の何千ものコミュニティメンバーに使用される可能性があります。ミッションクリティカルなシステムを支えるプロジェクトを改善することになるのです。

## OpenSearch へのコントリビューション領域

OpenSearch に意味のあるコントリビューションをするために、コーディングの達人である必要はありません。このプロジェクトでは、バックグラウンドやスキルレベルに関係なく、様々な参加方法が用意されています。

- **コードコントリビューション** は最も直接的な貢献方法です。バグ修正、小さな機能強化、既存コードのリファクタリングから始められます。自信がついてきたら、より大きな機能やパフォーマンス改善に取り組むこともできます。
- **バグ報告** は非常に重要です。OpenSearch を使用中に問題に遭遇した場合、詳細な手順、ログ、スクリーンショットとともに GitHub で報告することで、開発者が問題をより早く特定し解決できるようになります。
- **Issue のトリアージ** では、バグの検証、Issue の説明の改善、解決策の提案などを通じてメンテナーを支援できます。
- **コードレビュー** は、コントリビューションの全体的な品質を向上させるだけでなく、コードベースやプロジェクト標準への理解を深めることにもつながります。
- **機能テスト** を開発中やプレリリース段階で行うことで、最終製品を形作る貴重なユーザーフィードバックを提供できます。
- **ドキュメント作業** は非常に重視されています。既存ドキュメントの更新、チュートリアルの執筆、インストール手順の明確化などが含まれます。
- **ブログや動画** で自分の経験、解決策、ユースケースを共有することで、認知度を高め、他の人のコントリビューションを促進できます。

## OpenSearch Project GitHub の概要

[GitHub の OpenSearch Project](https://github.com/opensearch-project) にアクセスしてみてください。以下に関連する 130 以上のリポジトリがあります。

- OpenSearch Core – 検索エンジン本体 (Java ベース)
- OpenSearch Dashboards – エンジン管理用 UI (JavaScript/TypeScript)
- プラグイン – ML、セキュリティ、オブザーバビリティなど
- ユーティリティ – 共通ユーティリティ、OpenSearch、Python クライアントなど

興味とプログラミングスキルに基づいてリポジトリを選んでください。私は OpenSearch Core から始め、その後 ML 分野に踏み込むために ML Commons を探求しました。

## ステップバイステップ: 初めてのコントリビューション方法

### ステップ 1: GitHub のセットアップとリポジトリの選択

- まだ持っていない場合は、公開 GitHub アカウントを作成します。
- コントリビューションしたいリポジトリをフォークします (例: [https://github.com/opensearch-project/OpenSearch](https://github.com/opensearch-project/OpenSearch))。
- ローカルマシンにクローンします。
- リポジトリ内の以下の重要なファイルを確認します。
  - README.md、CONTRIBUTING.md、CODE_OF_CONDUCT.md
  - DEVELOPER_GUIDE.md

※コントリビューションがガイドラインに沿っており、所属組織によって制限されていないことを確認してください。

### ステップ 2: 開発環境のセットアップ

特定のリポジトリの開発者ガイド (DEVELOPER_GUIDE.md) に従ってください。OpenSearch Core の場合、通常以下が必要です。

- JDK
- Gradle
- Docker (統合テスト用)
- IntelliJ や Eclipse などの IDE

まず、コードを変更せずにビルドとテストを実行します。成功したら、ログの追加など小さな変更を試して、すべてが正しくセットアップされていることを確認します。

### ステップ 3: "good first issue" を見つける

GitHub の Issues タブで "good first issue" ラベルが付いた Issue を検索します。これらは新しいコントリビューターが簡単にオンボーディングできるよう厳選されています。作業を始める前に、担当者を確認しコメントを読んで、他の誰かがすでに取り組んでいないことを確認してください。

**成功のためのヒント: 私の経験から学んだこと**

適切な "good first issue" を選ぶのは大変かもしれませんが、いくつかの実践的な戦略に焦点を当てることで、私の旅はよりスムーズで実りあるものになりました。以下は、私が成功するのに役立ち、エコシステムに新しく参加する人にも役立つ重要な教訓です。

**テストのバグから始める**

OpenSearch への旅を始めたとき、私は意図的にテストのバグを探してコントリビューションしました。これらは通常、自己完結型でリスクが低く、複雑なビジネスロジックの変更を伴いません。そのため、プロジェクトの構造を理解するのに最適で、威圧感も少ないです。例えば、私の最初のコントリビューションは OpenSearch Core と ML Commons で、テストの問題を修正したりテストカバレッジを改善したりしました。この種のバグはレビューも早く、プロジェクトのワークフローやコーディング標準をまだ学んでいる段階で重要な早期フィードバックを得られます。テストのバグを修正することで自信がつき、リポジトリへの理解が深まりました。

GitHub 参照: [https://github.com/opensearch-project/ml-commons/issues/1382](https://github.com/opensearch-project/ml-commons/issues/1382)

**自分の専門分野で作業する**

自分の強みを活かすことは、早い段階で意味のあるコントリビューションをする素晴らしい方法です。私は OpenJDK のバックグラウンドがあるため、自分の経験に合った Issue、特にロケール処理や国際化に関連するものを探しました。そのようなタスクの 1 つは、非推奨のロケールプロバイダー Compat を Common Locale Data Repository (CLDR) に置き換えることでした。すでに知っていることを活用することで、立ち上がり時間を短縮し、インパクトを高められます。

GitHub 参照: [https://github.com/opensearch-project/OpenSearch/issues/11550](https://github.com/opensearch-project/OpenSearch/issues/11550)

**スタックトレースのある Issue を選ぶ**

GitHub の Issue にスタックトレースが含まれている場合、それは問題へのロードマップとして機能します。障害が発生した場所を、クラスやメソッドまで正確に教えてくれます。この種の Issue は、コードベースの関連部分にすぐにジャンプできるため、デバッグがはるかに簡単だと感じました。推測作業の多くを排除し、バグの再現と修正に集中できます。これは、コードベースのナビゲーションが困難な大規模プロジェクトで特に役立ちます。

GitHub 参照: [https://github.com/opensearch-project/ml-commons/issues/3669](https://github.com/opensearch-project/ml-commons/issues/3669)

**詳細に記述された Issue を探す**

一部のメンテナーは、バグレポートや機能リクエストを書く際に非常に丁寧です。問題を説明するだけでなく、有用なコンテキスト、潜在的な解決策、影響を受けるコードの部分に関するメモも提供してくれます。初心者にとって、これらは純金のようなものです。ミニチュートリアルとして機能し、問題を理解して修正するのにかかる時間を大幅に短縮できます。

印象的な例の 1 つは、`EncryptorImpl` の `initMasterKey()` というメソッドに関連するパフォーマンスの問題でした。レポートでは CountDownLatch の使用がボトルネックであることを明確に特定し、ActionListener への置き換えを提案していました。このようなガイダンスがあれば、修正がはるかに取り組みやすく、満足感のあるものになりました。

GitHub 参照: [https://github.com/opensearch-project/ml-commons/issues/3510](https://github.com/opensearch-project/ml-commons/issues/3510)

**プロのヒント:** 最初のいくつかのプルリクエスト (PR) では、複雑なバグを解決しようとするよりも、コントリビューションプロセスを学ぶことに集中してください。

作業する Issue を決めたら、コメントを残して自分にアサインしてください。これにより、他の人が知らずに同じ Issue に取り組むことを防げます。

### ステップ 4: 修正を行い PR を作成する

- main から新しいブランチを作成します。
- 変更を行いコミットします。
- ブランチをプッシュして PR を作成します。
- リポジトリの PR テンプレートに従い、CI チェックを待ちます。
- レビューコメントに対応し、必要に応じて PR を修正します。

## コミュニティとの関わり方

### Slack

参加する最良の方法の 1 つは、Slack の OpenSearch コミュニティに参加することです。Dashboards、Search、ML など、各主要コンポーネントには専用のチャンネルがあります。これらのスペースは、質問をしたり、実装の詳細を議論したり、進捗を共有したりするのに最適です。遠慮せずに連絡してください。メンテナーや他のコントリビューターは一般的に非常に歓迎的で親切で、特に新しいコントリビューターに対してそうです。これらのチャンネルで活動することで、学習曲線が加速するだけでなく、進行中の開発について最新情報を得られます。

Slack リンク: https://opensearch.org/slack.html

### トリアージミーティング

OpenSearch では、メンテナーが Issue をレビューし優先順位付けする定期的なトリアージミーティングを開催しています。これらのミーティングに参加することは、プロジェクトにとって何が重要かを把握する素晴らしい方法です。また、自己紹介をし、興味を表明し、どこから始めるべきかガイダンスを求める絶好の機会でもあります。メンテナーはこれらのミーティング中に初心者向けの Issue を提案することが多く、自分のスキルに合ったものを見つけやすくなります。

ミーティング招待リンク: [https://www.meetup.com/opensearch/events/](https://www.meetup.com/opensearch/events/)

### YouTube

OpenSearch の YouTube チャンネルには、チュートリアル、デモ、コミュニティトークなど、有用なコンテンツが満載です。これらの動画は、様々なコンポーネントがどのように機能するか、人々がどのように使用またはコントリビューションしているかについて深い洞察を提供します。これらを視聴することで、エコシステムをより良く理解できました。

YouTube チャンネルリンク: [https://www.youtube.com/@OpenSearchProject](https://www.youtube.com/@OpenSearchProject)

### ミートアップとカンファレンス

地元の OpenSearch コミュニティミートアップや OpenSearchCon イベントに参加することも、コントリビューターやメンテナーとつながる素晴らしい方法です。これらの対面イベントはリアルタイムのコラボレーションを促進し、ネットワークを構築し、プロジェクトへの関与を深めるのに役立ちます。

お近くのユーザーグループはこちらで見つけられます: https://opensearch.org/user-groups/

## 最後のステップ: 提出して関わり続ける

最初の PR を提出した後は、以下を行いましょう。

- GitHub でリポジトリをウォッチする
- 他の人の PR をレビューする
- コミュニティイベントに参加する
- ブログ投稿やイベントトークで学んだことを共有する

あなたのオープンソースの旅が正式に始まりました!

---

OpenSearch は単なるツールではなく、コミュニティです。テストのバグを修正するにしても、新機能を構築するにしても、あなたのコントリビューションは重要です。

**私からのアドバイス:**
小さく始めて、着実に成長し、助けを求めることを躊躇しないでください。コミュニティの誰もがあなたの成功を望んでいます。
