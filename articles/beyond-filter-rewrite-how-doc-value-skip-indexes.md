[以前のブログ記事](https://opensearch.org/blog/transforming-bucket-aggregations-our-journey-to-100x-performance-improvement/)では、フィルターリライトとマルチレンジトラバーサルが Lucene の BKD ツリーを活用して、date histogram アグリゲーションを最大 100 倍高速化した方法を紹介しました。これらの手法は非常に効果的でしたが、根本的な制約がありました。クエリフィルターとアグリゲーションフィールドが同一である必要があり、複雑なサブアグリゲーションロジックをサポートできませんでした。フィルターフィールドとアグリゲーションフィールドが異なる場合や、サブアグリゲーションがドキュメントごとの評価を必要とする場合、クエリエンジンは依然として一致するすべてのドキュメントを 1 つずつスキャンするフォールバック処理に頼っていました。

本記事では、Lucene 10 の doc value skip index を活用して、フィルターフィールドとアグリゲーションフィールドが無相関であっても最大 28 倍高速なアグリゲーションを実現した方法を紹介します。

## フィルターリライトの限界

フィルターリライトは、date histogram アグリゲーションを一連のレンジフィルターに変換し、Lucene の BKD ツリー (`Points` インデックス) を使用して、個々の doc value を参照せずにバケットごとのドキュメント数をカウントします。マルチレンジトラバーサルは、すべてのバケットをツリーの単一のソート済みパスで処理することで、これをさらに改善しました。

どちらの手法も、クエリフィルターで使用されるフィールドがアグリゲーション対象のフィールドと同一であるという前提に依存しています。以下のクエリは `trip_distance` でフィルターしつつ `dropoff_datetime` でアグリゲーションを行う例です。

```json
{
  "size": 0,
  "query": {
    "range": {
      "trip_distance": {
        "gte": 0,
        "lte": 20
      }
    }
  },
  "aggs": {
    "dropoffs_over_time": {
      "date_histogram": {
        "field": "dropoff_datetime",
        "calendar_interval": "month"
      }
    }
  }
}
```

`trip_distance` と `dropoff_datetime` は異なるフィールドであるため、`trip_distance` の BKD ツリーは `dropoff_datetime` の値がバケット間でどのように分布しているかについての情報を提供しません。エンジンはアグリゲーションフィールドに対するレンジフィルターへのリライトができず、従来のアプローチにフォールバックする必要があります。つまり、一致するすべてのドキュメントを反復処理し、各ドキュメントの `dropoff_datetime` の doc value を取得して、適切なバケットに配置します。

同様に、サブアグリゲーションがドキュメントごとの計算を必要とする場合 (例えば、各時間バケット内の平均メトリクスの計算)、フィルターリライトはドキュメント数のみを提供し、個々のフィールド値へのアクセスを提供しないため、役に立ちません。

これらはエッジケースではありません。実際には、多くの分析クエリが 1 つのディメンションでフィルターしながら別のディメンションでアグリゲーションを行い、サブアグリゲーションはダッシュボードやレポートで一般的に使用されています。より汎用的なアプローチが必要でした。

## Doc value skip index: アグリゲーションエンジンの新しいツール

Lucene 10.0 から、新しいインデックス構造が利用可能になりました。数値型 doc value の上に構築されるオプションの skip index で、[PR #13449](https://github.com/apache/lucene/pull/13449) で導入され、[PR #13563](https://github.com/apache/lucene/pull/13563) でマルチレベルサポートが追加されました。この構造は `DocValuesSkipper` 抽象化を通じて公開され、本記事で説明する最適化の基盤を提供します。

### Skip index とは

一般的なコンピュータサイエンスにおいて、スキップリストはソート済みデータの上に複数レベルのリンクリストを重ね、ランダム化によってどの要素が上位レベルに現れるかを決定する確率的データ構造です。これにより、期待計算量 O(log n) の検索時間が実現されます。

Lucene の doc value skip index はマルチレベルの概念を借用していますが、完全に決定的です。ランダム化を使用する代わりに、ドキュメントを固定サイズの区間に分割し、セグメント作成時のコンパイル時にサマリーレベルの階層を構築します。ランダム性は一切関与せず、構造はドキュメント数と設定された区間サイズによって完全に決定されます。

これは本の目次のようなものです。必要な情報を見つけるためにすべてのページをスキャンする代わりに、まず章の見出しを確認し、次にセクションの見出しを確認し、正確なページに絞り込みます。Skip index も同じように機能します。アグリゲーションエンジンが個々の値を調べるかどうかを決定する前に、大きなドキュメントブロックのサマリーメタデータを確認できるようにします。

### Lucene における skip index の実装

Lucene の実装は、最大 4 レベル (レベル 0〜3) の階層構造を使用します。ベースレベル (レベル 0) は 4,096 ドキュメントの区間でサマリーを作成します。各上位レベルは下位レベルの 8 区間を集約します (2^3 のシフト)。以下の表に示します。

| レベル | 区間あたりのドキュメント数 | 説明 |
|--------|--------------------------|------|
| 0 | 4,096 | ベース区間 |
| 1 | 32,768 (4,096 × 8) | レベル 0 の 8 区間ごと |
| 2 | 262,144 (4,096 × 64) | レベル 1 の 8 区間ごと |
| 3 | 2,097,152 (4,096 × 512) | レベル 2 の 8 区間ごと |

最悪ケースとして 2^31 − 1 (約 21 億) ドキュメントのインデックスでは、skip index 階層はレベル 0 で約 524,288 エントリ、レベル 1 で 65,536、レベル 2 で 8,192、レベル 3 で 1,024 を含みます。したがって、検索空間は数十億のドキュメントから数千のメタデータチェックに削減されます。

各区間エントリはコンパクト (わずか 29 バイト) で、以下のメタデータをエンコードします。

1. **レベル数** (1 バイト): このエントリが含まれるレベルの数。
2. **最小値と最大値** (16 バイト): この区間内のフィールド値の範囲。
3. **最小と最大のドキュメント ID** (8 バイト): この区間のドキュメント ID の境界。
4. **ドキュメント数** (4 バイト): この区間内のドキュメント数。

このメタデータにより、アグリゲーションエンジンは各区間に対して 3 つの判断を行えます。

1. 区間の最小/最大値が現在のアグリゲーションバケットの**完全に外側**にある場合、区間全体をスキップする。
2. 区間の最小/最大値が単一のバケットに**完全に収まる**場合、個別の検査なしにすべてのドキュメントを一括カウントする。
3. 区間が**複数のバケットにまたがる**場合、ドキュメントを個別に処理するフォールバックを行う。

以下の図に示すように、トラバーサルは利用可能な最上位レベルから開始し、必要な場合にのみ下位に降りていきます。これは以前の最適化における BKD ツリートラバーサルがサブツリー全体をスキップする方法と似ています。主な違いは、この構造が `Points` インデックスではなく doc value 上で動作するため、クエリがどのフィールドでフィルターしているかに関係なく機能することです。

![](/images/beyond-filter-rewrite-how-doc-value-skip-indexes/0dd079aa4ae9.png)
*図 1: Skip index の可視化。各レベルの最小/最大メタデータに基づいて、ドキュメントの範囲をスキップまたは一括カウントできる様子を示します。*

## Skip index がフィルターリライトの制約をどのように解決するか

先に特定した 2 つの主な制約を振り返ります。

1. **無相関フィールド**: フィルターリライトはフィルターフィールドとアグリゲーションフィールドが同一である必要がある。
2. **複雑なサブアグリゲーション**: フィルターリライトはカウントのみを提供し、バケット内のドキュメントごとの計算をサポートできない。

Skip index は、アグリゲーションフィールドの doc value 上で直接動作し、一致するドキュメントセットがどのように生成されたかに依存しないため、両方の制約に対処します。クエリエンジンはまずクエリを評価し (フィルターフィールドに適したインデックスを使用)、一致するドキュメント ID のセットを生成します。次に、アグリゲーション時にアグリゲーションフィールドの skip index を参照して、一致するドキュメントのブロックをスキップまたは一括カウントできるかどうかを判断します。

この分離が重要な洞察です。Skip index はドキュメントセットがフィルターされたかどうかを気にしません。各区間内のアグリゲーションフィールドの値分布に関する情報のみを必要とします。

### 例

例えば、`process.name` でフィルター (term フィルター) し、`@timestamp` でアグリゲーション (日単位の date histogram) するクエリを考えます。

```json
GET /logs-*/_search
{
  "query": {
    "bool": {
      "must": [
        {
          "range": {
            "@timestamp": {
              "gte": "2024-01-01",
              "lte": "2024-01-31"
            }
          }
        },
        {
          "term": {
            "process.name": "systemd"
          }
        }
      ]
    }
  },
  "aggs": {
    "daily_counts": {
      "date_histogram": {
        "field": "@timestamp",
        "calendar_interval": "day"
      }
    }
  }
}
```

Skip index がない場合、`@timestamp` に対する range クエリはフィルターリライトの恩恵を受ける可能性がありますが、`process.name` に対する `term` フィルターの存在によりクエリのリライトが妨げられます。エンジンは両方の条件に一致するすべてのドキュメントを反復処理し、各 `@timestamp` の値を個別に取得する必要があります。

`@timestamp` で skip index が有効な場合、アグリゲーションエンジンは収集時に skip index を参照できます。最小と最大の `@timestamp` 値が同じ日次バケットに収まる区間に遭遇した場合、個々の値を参照せずにその区間内のすべての一致するドキュメントを一括カウントします。以下の図に示します。

![](/images/beyond-filter-rewrite-how-doc-value-skip-indexes/70adccd264b8.svg)
*図 2: Skip index が有効な場合、エンジンが最小/最大値が同じバケットに収まる区間に到達すると (例えば、14:00:01 から 14:20:59 の値を持つ Block 42)、ブロック全体を一括カウントします。このプロセスはバケット境界をまたぐブロック (例えば、翌日に跨る) に到達するまで連続するブロックで繰り返され、その時点で個別の doc value 収集にフォールバックします。これにより、ルックアップの総数を約 85% 削減できます。*

## popcount による効率的なカウント

一括カウントのステップには、検討に値する微妙な点があります。Skip index が区間内のすべての値が単一のバケットに収まることを示す場合、一括カウントが可能であることはわかりますが、区間のドキュメント数を直接使用することはできません。Skip index はセグメント内のすべてのドキュメントをカバーしますが、クエリによって生成される `DocIdSetIterator` はクエリフィルターに一致したドキュメントのサブセットのみを表します。フィルターされたサブセットのうち、skip index 区間内に含まれるドキュメント数をカウントする必要があります。

1 つのアプローチは、イテレーターを 1 ドキュメントずつ進めてカウントすることです。しかし、これではスキップの目的の大部分が失われます。代わりに、ハードウェアレベルの最適化を使用します。`popcount` (ポピュレーションカウント) CPU 命令は、マシンワード内のセットビット数を単一の操作でカウントします。

クエリエンジンが `DocIdSetIterator` を生成する際、一致するドキュメントセットは内部的にビットセットとして表現されることが多く、各ビットがドキュメント ID に対応し、ドキュメントがクエリに一致した場合に `1` に設定されます。Skip index 区間内に含まれる一致ドキュメント数をカウントするには、このビットセットの関連部分 (区間の最小/最大ドキュメント ID 範囲内のドキュメント ID に対応するビット) を抽出し、それらのワードに対して `popcount` を適用します。これにより、個別に反復処理することなく、区間内の一致ドキュメントの正確な数が得られます。

`DocIdSetIterator` は一括収集インターフェースもサポートしています。`nextDoc()` を繰り返し呼び出す代わりに、コレクターはドキュメント ID のストリームを受け取ることができます。ストリーム内のすべてのドキュメント ID が同じバケットに収まる場合 (skip index によって判断)、ストリーム全体を 1 回の操作で収集できます。バケット判定のための skip index メタデータと、フィルターされたセット内の効率的なカウントのための `popcount` の組み合わせが、この最適化を実用的にしています。予備的なベンチマークでは、前述の skip index の改善に加えて、このアプローチにより最大 3 倍の追加改善が示されています。

## ソート済みデータ: Skip index が最も効果を発揮する場面

Skip index は、ドキュメントがアグリゲーションフィールドでソートされている場合に最高のパフォーマンスを発揮します。データがソートされている場合、連続するドキュメントは類似した値または単調増加する値を持つため、各 4,096 ドキュメント区間内の最小/最大範囲が狭くなります。狭い範囲は単一のアグリゲーションバケットに完全に収まる可能性が高く、一括カウントの機会を最大化します。

時系列ワークロード (ログ分析、メトリクス、オブザーバビリティ) では、タイムスタンプフィールドが自然に適合します。ほとんどの時系列データはおおよそ時系列順に到着し、OpenSearch のデータストリームはこの順序を維持します ([Data streams](https://docs.opensearch.org/latest/im-plugin/data-streams/) を参照してください)。

### データのソート順を維持する

単にタイムスタンプフィールドがあるだけでは、セグメントのマージやドキュメントの追加に伴ってデータがソートされた状態を維持することは保証されません。ソート順を維持するには 2 つのアプローチがあります。

1. **インデックスソート (推奨)**: セグメントのマージに関係なくデータがソートされた状態を維持するために、明示的なインデックスソート設定を構成します。

```json
{
  "settings": {
    "index.sort.field": "@timestamp",
    "index.sort.order": "desc"
  }
}
```

これにより、すべてのセグメントがタイムスタンプ順を維持することが保証され、一貫した skip index パフォーマンスが提供されます。

2. **ログマージポリシー**: あるいは、受信ドキュメントの順序を保持するマージポリシーを使用します。`log_byte_size` マージポリシーは、時系列データに典型的な追記専用ワークロードの時間的順序を維持する傾向があります。

データがソートされていない場合、skip index は正しく機能しますが、各区間の最小/最大範囲が複数のバケットにまたがる可能性が高いため、スキップと一括カウントの機会は少なくなります。

## Skip index の有効化

Skip index は、OpenSearch のバージョンに応じて自動的に、またはマニュアル設定で有効化できます。

### バージョンごとのデフォルト動作

OpenSearch は skip index のサポートを段階的に拡大しています。以下の表に示します。

| バージョン | 動作 |
|-----------|------|
| OpenSearch 3.2 | 数値フィールドに `skip_list` マッピングパラメーターを導入 (デフォルト: `false`) |
| OpenSearch 3.3 | `@timestamp` という名前の date フィールドに対する date histogram アグリゲーションで skip index を自動的に有効化 |
| OpenSearch 3.4 | `@timestamp` に対する auto date histogram アグリゲーションにも自動 skip index 最適化を拡張 |

詳細については、[Date field type](https://docs.opensearch.org/latest/mappings/supported-field-types/date/) を参照してください。

### マニュアル設定

他の数値フィールドで skip index を有効にするには、フィールドマッピングに `skip_list` パラメーターを追加します。

```json
PUT /my-index
{
  "mappings": {
    "properties": {
      "custom_timestamp": {
        "type": "date",
        "skip_list": true
      },
      "price": {
        "type": "long",
        "skip_list": true
      }
    }
  }
}
```

## パフォーマンス結果

OpenSearch Benchmark を使用して `http_logs` および `big5` データセットでパフォーマンスを測定しました。結果は、特にフィルターリライトが以前は役に立たなかったクエリで大幅な改善を示しています。

### Date histogram のパフォーマンス (http_logs ワークロード)

以下の表は、`http_logs` データセットを使用した [PR #19130](https://github.com/opensearch-project/OpenSearch/pull/19130) のテストに基づく、ベースラインと skip index のパフォーマンスの比較です。

| オペレーション | ベースライン (p90) | Skip index あり (p90) | 改善 |
|--------------|-------------------|----------------------|------|
| `hourly_agg_with_filter` | 998 ms | 36 ms | **28 倍高速** |
| `hourly_agg_with_filter_and_metrics` | 1,618 ms | 970 ms | **40% 高速** |

最初のオペレーションは、無相関フィルターを持つ純粋なカウントアグリゲーションに対する skip index の完全な効果を示しています。2 番目のオペレーションにはサブアグリゲーション (メトリクス) が含まれており、メトリクスフィールドのドキュメントごとのメトリクス計算には依然として個別の値参照が必要なため、改善は小さくなっています。

スループットは date histogram クエリで 21% 改善し、インデックスパフォーマンスへの測定可能な影響はありませんでした。

### Auto date histogram のパフォーマンス (big5 ワークロード)

OpenSearch 3.4 では、skip index の最適化が auto date histogram アグリゲーションにも拡張されました。以下の表は、`big5` データセットを使用した [PR #20057](https://github.com/opensearch-project/OpenSearch/pull/20057) のテストに基づく、ベースラインと skip index のパフォーマンスの比較です。

| オペレーション | ベースライン (p90) | Skip index あり (p90) | 改善 |
|--------------|-------------------|----------------------|------|
| `range-auto-date-histo` | 2,099 ms | 324 ms | **6.5 倍高速** |
| `range-auto-date-histo-with-metrics` | 5,733 ms | 3,928 ms | **35% 高速** |

これらの結果は、range クエリに対するフィルターリライト最適化と auto date histogram に対する skip index を組み合わせたもので、2 つの手法がどのように補完し合うかを示しています。

### インデックスサイズへの影響

Skip index のストレージオーバーヘッドは最小限です。以下の表に示します。

| 設定 | サイズ増加 |
|------|----------|
| `@timestamp` フィールドのみ | 約 0.1% |
| すべての数値フィールド | 約 1% (例: big5 で 22 GB → 23 GB) |

この最小限のオーバーヘッドのため、OpenSearch はデフォルトで `@timestamp` フィールドのみに skip index を有効にし、大きなストレージコストなしにターゲットを絞った効果を提供しています。

## Skip index が以前の最適化をどのように補完するか

Skip index はフィルターリライトやマルチレンジトラバーサルを置き換えるものではなく、補完するものです。以下の表に各手法が適用される場面をまとめます。

| 手法 | 適用される場面 | 制約 |
|------|--------------|------|
| フィルターリライト | フィルターフィールド = アグリゲーションフィールド、単純なカウント | 無相関フィールドやサブアグリゲーションを処理できない |
| マルチレンジトラバーサル | フィルターリライトと同じ条件で、多数のバケットがある場合 | フィルターリライトと同じ制約 |
| Skip index | 任意のクエリパターン、任意のアグリゲーションフィールド | ソート済みデータで最も効果的。ソートされていないデータでは緩やかにフォールバック |

クエリがフィルターリライトの対象となる場合、OpenSearch は doc value へのアクセスを完全に回避できるフィルターリライトを使用します。フィルターリライトが適用できない場合、skip index がすべてのドキュメントをスキャンするよりも大幅に高速なフォールバックを提供します。2 つのアプローチはクエリ空間の補完的な部分をカバーしています。

## 今後の予定

OpenSearch コミュニティは skip index の機能を積極的に拡張し、補完的な最適化を探求しています。

- **Min/max アグリゲーション**: ドキュメントを一切参照せずに skip index メタデータを使用して即座に min/max を計算する ([issue #20174](https://github.com/opensearch-project/OpenSearch/issues/20174))。
- **拡張バケット処理**: より複雑なアグリゲーションパターンのために、複数の所有バケットオーディナルをサポートする。
- **一括処理とベクトル化**: Lucene での最近の取り組みは、ドキュメントを 1 つずつではなくバッチで処理するように収集ループを再構築することに焦点を当てています。フィールド値を連続バッファにロードしてバルクで操作することで、JVM は自動ベクトル化と SIMD 命令を適用して、多くの値に対するスカラー関数 (合計、平均、min/max) を同時に計算できます。このバッチ指向のアプローチは、仮想メソッド呼び出しのオーバーヘッドも削減します。ドキュメントごとに仮想 `collect()` メソッドを呼び出す代わりに、バッチごとに 1 回呼び出すことで、JVM の JIT コンパイラーがホットパスをより効果的にインライン化できます。これらの手法は、各バケット内でメトリクスを計算するサブアグリゲーションで特に有望であり、現在はドキュメントごとのオーバーヘッドが支配的です。

この取り組みは、以前の記事で説明した反復的なアプローチを継続しています。ボトルネックを特定し、基盤となるデータ構造を理解し、不要な作業を回避する方法を見つけます。フィルターリライトは、インデックスツリーを通じてドキュメントをカウントする方が doc value を反復処理するよりも高速であることを教えてくれました。Skip index はその教訓を拡張します。doc value を使用する必要がある場合でも、すべての値を 1 つずつ参照することを回避できます。

詳細を確認したり貢献したりするには、skip index 実装計画の完全な [メタ issue #18882](https://github.com/opensearch-project/OpenSearch/issues/18882) と、ベンチマーク結果および進行中の作業の [issue #19384](https://github.com/opensearch-project/OpenSearch/issues/19384) を参照してください。
