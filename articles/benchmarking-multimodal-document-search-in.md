実世界の多くの文書（財務報告書、研究論文、技術レポートなど）には、テキスト以外の要素も含まれています。表、グラフ、図解、画像などに重要な情報が含まれているのです。従来のテキストベースの検索では、こうした視覚的なコンテンツを捉えることはできません。

この問題を解決するため、OpenSearch ではベクトル検索とニューラル検索パイプラインを使用して実装されたマルチモーダル文書検索をサポートしています。ただし、マルチモーダルデータのインデックスとクエリにはさまざまなアプローチがあり、それぞれ検索品質、レイテンシー、複雑さの面で異なるトレードオフがあります。

ユースケースに最適なアプローチを見つけられるよう、同じ 1,000 ページのレポートデータセットで 3 つのアプローチをベンチマークしました。本記事では各アプローチを説明し、ベンチマーク結果を提示し、それぞれをいつ使うべきかを解説します。

**TL;DR:** OpenSearch で 1,000 ページのレポートを対象に 3 つのマルチモーダル検索アプローチをベンチマークしました。

- ColPali は再現率 92% と品質がリードするが、レイテンシーは最大
- BDA モダリティ対応埋め込みは品質と速度のバランスが良い
- テキストのみのチャンキングは最もシンプルだが、視覚的コンテンツを取りこぼす
- グラフ、表、図解を含む文書にはマルチモーダル検索が必要
- 視覚要素が回答を担う場合、テキストのみのアプローチは機能しない
- 品質とレイテンシーの優先度に応じて選択する

## データセット

ベンチマークには、Hugging Face の [vidore/syntheticDocQA_government_reports_test](https://huggingface.co/datasets/vidore/syntheticDocQA_government_reports_test) データセットを使用しました。このデータセットには、1,000 ページのスキャンされたレポートと 100 組のクエリ・回答ペアが含まれています。各クエリには関連文書が 1 つだけ対応しており、クリーンな検索ベンチマークとなっています。

## 比較した 3 つのアプローチ

以下の 3 つのアプローチを評価しました。それぞれが品質とレイテンシーのスペクトル上で異なる位置を占めています。

### アプローチ 1: ColPali レイトインタラクション再ランキング

ColPali は、各文書ページを画像として扱い、パッチ単位の複数の埋め込みを生成するビジョン言語モデルです（*レイトインタラクション* アプローチ）。検索時にはクエリも複数のトークン埋め込みにエンコードされ、すべてのクエリトークンとすべての文書パッチを比較することで関連度が計算されます。これは ColBERT がテキスト向けに行う手法を、視覚コンテンツに拡張したものと言えます。

以下の図に ColPali のプロセスを示します。

```
文書画像 → SageMaker GPU エンドポイント (ColPali モデル) → 複数のパッチ埋め込みを OpenSearch に保存 → 検索時、レイトインタラクションスコアリングによってクエリ埋め込みをすべてのパッチ埋め込みと比較
```

このアプローチでは、ページ全体（テキスト、表、グラフ、レイアウトを含む）を単一の画像として処理し、テキスト抽出やパースを必要としません。

### アプローチ 2: BDA モダリティ対応埋め込み

このアプローチでは、Amazon Bedrock Data Automation (BDA) を使用して、各文書を型付きの要素（段落、表、図）にパースします。その後、各要素は Amazon Titan Multimodal Embeddings を使用して個別に埋め込まれます。テキスト要素はテキストモダリティで、図は画像モダリティ（元のページからバウンディングボックス座標を使用して切り出したもの）で埋め込まれます。すべての埋め込みは同じ 1,024 次元のベクトル空間に保存されます。

以下の図に BDA のプロセスを示します。

```
文書 → BDA によるパース → 要素抽出 (PARAGRAPH、TABLE、FIGURE) → テキスト要素には Titan テキスト埋め込み、図要素には Titan 画像埋め込みを適用 → すべてをネストされたドキュメントとして OpenSearch に保存 → クエリ時にニューラル検索
```

このアプローチは、モダリティ固有の情報を保持します。たとえばグラフは、OCR (Optical Character Recognition) テキストではなく画像として埋め込まれます。

### アプローチ 3: テキストのみのチャンキング

テキストのみのチャンキングは最もシンプルなアプローチであり、実務でよく見られるパターンを表しています。まず、BDA が文書から生のテキストを抽出します。次に、テキストを約 200 語のセグメントに 30 語のオーバーラップでチャンキングし、各チャンクを Amazon Titan Text Embeddings モデルを使用して埋め込みます。

以下の図にテキストのみのチャンキングのプロセスを示します。

```
文書画像 → BDA によるテキスト抽出 → 約 200 語のセグメントにチャンキング → チャンクごとに Titan テキスト埋め込み → ネストされたドキュメントとして OpenSearch に保存 → クエリ時にニューラル検索
```

このアプローチは実装が容易ですが、視覚情報をすべて捨ててしまいます。表は平坦なテキストになり、グラフ、図解、その他の視覚要素は完全に無視されます。

## ベンチマーク構成

公平な比較を保つため、ベンチマーク中は以下の変数を制御しました。

| 変数 | 値 |
|------|-----|
| データセット | 同一の 1,000 ページレポート |
| クエリ | 同じ 100 クエリ（グラウンドトゥルースの関連性情報付き） |
| OpenSearch クラスター | 同一のローカルクラスター (OpenSearch 3.5.0) |
| 評価指標 | k=5 および k=10 での NDCG、MRR、Recall |
| 実行方法 | インターリーブ（順序バイアスを避けるため、クエリをアプローチ間でローテーション） |
| ウォームアップ | 計測前にアプローチごとに 3 クエリのウォームアップを実施 |

## 評価指標の定義

検索品質とパフォーマンスを評価するため、以下の指標を使用しました。

- **NDCG@k (Normalized Discounted Cumulative Gain)**: 関連結果が上位にどれだけうまくランキングされているかを測定します。スコア 1.0 は完璧なランキングを意味します。@5 と @10 は、上位 5 件または 10 件を対象に評価することを示します。
- **MRR@k (Mean Reciprocal Rank)**: 最初の関連結果がどれだけ早く出現するかを測定します。スコア 1.0 は、正しい文書が常に 1 位にランクされることを意味します。
- **Recall@k**: 上位 *k* 件の結果の中で取得できた関連文書の割合を測定します。
- **Latency (ms)**: クエリあたりの平均検索応答時間をミリ秒単位で示します。
- **P95 latency (ms)**: 95 パーセンタイル応答時間を示します。クエリの 95% はこの値より速く完了します。

## 検索品質の結果

以下の表に、検索品質指標とクエリレイテンシーについて 3 つのアプローチを比較した結果を示します。

| アプローチ | NDCG@5 | NDCG@10 | MRR@5 | Recall@5 | Recall@10 | Latency (ms) | P95 (ms) |
|------------|--------|---------|-------|----------|-----------|--------------|----------|
| ColPali レイトインタラクション再ランキング | 0.77 | 0.77 | 0.72 | 0.92 | 0.92 | 1,683 | 1,969 |
| BDA モダリティ対応埋め込み | 0.51 | 0.54 | 0.45 | 0.69 | 0.77 | 257 | 306 |
| テキストのみのチャンキング | 0.38 | 0.42 | 0.33 | 0.52 | 0.64 | 294 | 370 |

ColPali はすべての品質指標でリードしています。NDCG@10 は 0.77、Recall@10 は 0.92 であり、92% の確率で上位 10 件の中から正しい文書を見つけ出します。また、高い NDCG@10 スコアは、関連文書がより上位のポジションに出現する傾向にあることを示しています。トレードオフはレイテンシーです。クエリあたり約 1,683 ms で、他の 2 つのアプローチと比較しておよそ 6～7 倍遅くなります。これは、レイトインタラクションスコアリングがクエリトークン埋め込みをすべての保存済みパッチ埋め込みと比較すること、および再ランキングステップが SageMaker GPU エンドポイントで実行されるためです。

BDA モダリティ対応埋め込みのアプローチは、他の 2 つの中間に位置します。中程度の検索品質 (NDCG@10 = 0.54) と低レイテンシー（約 257 ms）を提供します。画像と表を別々のモダリティとして保持するため、テキストのみのアプローチが見落とす情報を捉えることができます。

テキストのみのチャンキングは実装が最速ですが、スコアは最低 (NDCG@10 = 0.42) でした。文書に回答を担う表、グラフ、図解が含まれる場合、テキストのみのアプローチではそれらを取得できません。

## インジェストレイテンシーの結果

各文書のインジェストにかかる時間（50 文書のサンプル）も測定しました。以下の表にベンチマーク結果を示します。

| アプローチ | 平均 (秒/文書) | 中央値 (秒/文書) | P95 (秒/文書) |
|------------|----------------|------------------|----------------|
| ColPali レイトインタラクション再ランキング | 9.0 | 9.1 | 10.0 |
| BDA モダリティ対応埋め込み | 5.1 | 5.0 | 6.9 |
| テキストのみのチャンキング | 11.0 | 6.5 | 21.7 |

BDA モダリティ対応埋め込みのアプローチは、最も安定していて高速なインジェスト時間を示しました。ColPali レイトインタラクション再ランキングは、Amazon SageMaker の GPU 推論時間に左右され、文書あたり約 9 秒で安定しています。テキストのみのチャンキングは分散が最大です。中央値は 6.5 秒ですが、P95 は 21.7 秒まで上昇します。これは、一部の文書が多数のテキストチャンクを生成し、それぞれが個別の埋め込み呼び出しを必要とするためです。

## どのアプローチを選ぶべきか

最適なアプローチは、対象の文書と優先事項によって異なります。

- **ColPali レイトインタラクション再ランキング**: 検索品質を最優先し、高めの検索レイテンシーを許容できる場合は、このアプローチを選択してください。文書が視覚的に豊富（グラフ、図解、複雑なレイアウト）で、精度が速度より重要な場合に最適です。このアプローチには GPU エンドポイント（例: Amazon SageMaker）が必要です。
- **BDA モダリティ対応埋め込み**: 品質と低レイテンシーのバランスを求める場合は、このアプローチを選択してください。テキストと画像を別々のモダリティとして扱うため、混在コンテンツを含む文書をうまく処理できます。ほとんどのユースケースに適したバランスを提供します。
- **テキストのみのチャンキング**: 文書が主にテキストベースである場合、またはシンプルさと迅速な実装が優先事項の場合は、このアプローチを選択してください。セットアップが最も簡単ですが、視覚要素が重要な情報を担う文書では性能が劣ります。
