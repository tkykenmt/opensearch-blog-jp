[前回のブログ記事](https://opensearch.org/blog/introducing-agentic-search-in-opensearch-transforming-data-interaction-through-natural-language/)では、自然言語を使用してデータとやり取りできる OpenSearch のエージェンティック検索を紹介しました。会話形式で情報を検索できる機能は、情報検索を簡素化する画期的なステップであり、クエリの専門家でなくても必要な情報を正確に検索しやすくなります。大規模言語モデル (LLM) の柔軟性により、エージェンティック検索は適応性が高く、アグリゲーションやフィルタリングを必要とする分析的な質問から、意味的に類似したドキュメントを見つけることを目的とした検索質問まで、幅広い質問を理解して回答できます。自然言語で OpenSearch とやり取りする際、実際に実行されるクエリは見えません。そのため、多くの異なるユースケースにわたって検索システムを徹底的にテストしています。

本記事では、エージェンティック検索のパフォーマンスについて、2 つの重要な側面にわたる評価方法と結果を共有します。

- **検索関連性**: エージェンティック検索は関連するドキュメントをどの程度適切にランク付けするか。
- **実行精度**: エージェンティック検索は与えられた自然言語の質問に対して正しい構造化クエリを生成するか。

これらの側面を合わせることで、構造化検索タスクと非構造化検索タスクの両方にわたるエージェンティック検索の能力について包括的な知見が得られます。

## 検索関連性

検索関連性は、返されたドキュメントがどの程度適切にランク付けされているかを示します。ランク付き検索品質の測定には、標準的な評価指標である NDCG@10 (Normalized Discounted Cumulative Gain at rank 10) を使用しました。

### データセット

検索関連性を評価するために、確立された情報検索ベンチマークである BEIR (Benchmarking Information Retrieval) と BRIGHT (Bridging Retrieval and Insights through Grounding in Hard Tasks) データセットを使用しました。

#### BEIR データセット

[BEIR](https://github.com/beir-cellar/beir) は、情報検索モデルの評価に広く採用されている異種ベンチマークです。多様なドメインとタスクタイプにまたがり、検索の汎化性能を堅牢にテストできます。7 つの BEIR データセットを使用して検索関連性を評価しました。

1. **Natural Questions (NQ)**: 実際の Google 検索クエリと Wikipedia の回答のペア。
2. **FiQA**: 金融に関する意見の質問応答。
3. **SciFact**: 科学的主張の検証。
4. **SciDocs**: 科学文献における引用予測。
5. **TREC-COVID**: COVID-19 関連の情報検索。
6. **NF-Corpus**: 栄養と健康に関する情報検索。
7. **ArguAna**: 反論生成のための議論検索。

#### BRIGHT データセット

[BRIGHT](https://brightbenchmark.github.io/) は、従来のキーワードマッチングでは不十分な、複雑で推論集約型のクエリを使用して検索システムをテストするために設計された、より新しく大幅に難易度の高いベンチマークです。6 つの BRIGHT データセットを使用して検索関連性を評価しました。

1. **Biology**: 複雑な生物学的推論の質問。
2. **Earth science**: 地球科学の推論クエリ。
3. **Economics**: 経済分析の質問。
4. **LeetCode**: プログラミング問題の検索。
5. **Psychology**: 心理学研究の質問。
6. **StackOverflow**: 文脈理解を必要とする技術的なプログラミングの質問。

BRIGHT のクエリは、標準的な検索手法では困難になるよう特別に設計されています。表面的なキーワードの単純なマッチングではなく、質問の背後にある意図の理解が必要になることが多いためです。これにより、BRIGHT はエージェンティック検索のクエリ再構成機能が意味のある優位性を提供するかどうかを評価するのに理想的なデータセットとなっています。

### 評価セットアップ

13 のデータセットにわたって 6 つの検索構成を比較しました。フローエージェントには Anthropic Claude Opus 4.6 を使用し、ドキュメントテキストのベクトル埋め込み生成には `msmarco-distilbert-base-tas-b` モデルを使用しました。

```json
{
  "name": "huggingface/sentence-transformers/msmarco-distilbert-base-tas-b",
  "version": "1.0.3",
  "model_format": "TORCH_SCRIPT"
}
```

3 つの標準ベースライン手法をテストし、3 つのエージェンティック検索バリアントと比較しました。

- **標準 (ベースライン) 手法**:
  - **Lexical (BM25)**: 従来のキーワードベースの検索。
  - **Neural**: 密なエンベディングを使用したセマンティック検索。
  - **Hybrid**: Lexical と Neural の検索の組み合わせ。
- **エージェンティック手法**:
  - **Agentic (lexical)**: BM25 を基盤の検索手法として使用。
  - **Agentic (neural)**: Neural 検索を使用。
  - **Agentic (hybrid)**: Hybrid 検索を使用。

### システムプロンプトチューニング

システムプロンプトチューニングは、エージェンティック検索における重要な側面です。エージェントが特定のデータセットに対する検索の理解と計画を改善するために動作を調整できます。関連性評価では、BEIR と BRIGHT データセットのシンプルなインデックスマッピングのため、構造化データクエリではなくドキュメント検索に明示的に最適化された、検索モード固有のシステムプロンプトを `QueryPlanningTool` 用に作成しました。

検索手法ごとに最適なクエリフォーマットが異なります。Lexical 検索はキーワード拡張の恩恵を受け、Neural 検索は自然言語の記述でより良く機能します。各手法の強みを活かすクエリを生成するため、検索モード (Lexical、Neural、Hybrid) ごとに 3 つの異なるプロンプト戦略を設計しました。

#### Lexical 検索プロンプト

BM25 ベースの検索では、システムプロンプトがエージェントに対して、ユーザーのクエリを広範でキーワードが豊富な検索クエリに拡張するよう指示します。プロンプトは 4 ステップのプロセスに従います。

1. **理解**: コアトピック、情報タイプ、主要エンティティ、暗黙の制約を特定。
2. **強化**: ドメイン固有の用語、固有名詞、技術用語、略語の展開を追加。
3. **拡張**: 同義語、代替表現、共起語彙を追加。
4. **構成**: すべてを単一のフラットなキーワードバッグに結合 (自然言語の文ではなく、スペースで区切られた関連用語のみ)。

例えば、クエリ "what is the meaning of the dragon boat festival" は以下のテキストに拡張されます。
`"dragon boat festival meaning significance origin history tradition Chinese Duanwu Festival rice dumplings zongzi Qu Yuan poet commemoration fifth month lunar calendar racing paddling celebration cultural heritage dragon boats race competition memorial purpose symbolism customs rituals holiday traditional"`
エージェントは結果を `text` と `title` フィールドにわたる `multi_match query` として返します。

```json
{
    "query": {
        "multi_match": {
            "query": "<expanded keyword bag>",
            "type": "best_fields",
            "fields": ["text_key", "title_key"],
            "tie_breaker": 0.5
        }
    }
}
```

このアプローチは BM25 の用語マッチングメカニズムをターゲットにしています。関連する用語、同義語、ドメイン語彙でクエリを強化することで、BM25 が関連ドキュメントを検索する可能性を高めます。

#### Neural 検索プロンプト

Neural 検索では、戦略が根本的に異なります。密なエンベディングモデルは個々のキーワードではなく意味をエンコードするため、フラットなキーワードバッグの使用は逆効果になります。代わりに、システムプロンプトはエージェントに対して、完全なセマンティックインテントを捉える適切な自然言語の文としてクエリを書き直すよう指示します。

1. **理解**: コアトピック、情報タイプ、主要エンティティ、暗黙の制約を特定。
2. **強化**: ドメイン固有のコンテキストを追加し、略語を展開し、関連概念を組み込む。
3. **言い換え**: エンベディングモデルが意味的に類似したドキュメントを見つけるために使用できる、一貫性のある記述的な文章 (キーワードリストではない) を作成。
4. **焦点の維持**: すべての詳細が元のクエリに関連していることを確認し、脱線するトピックを避ける。

同じ dragon boat festival のクエリは以下のテキストに拡張されます。
`"The meaning and cultural significance of the Dragon Boat Festival (Duanwu Festival), a traditional Chinese holiday celebrated on the fifth day of the fifth lunar month, including its origins commemorating the poet Qu Yuan, the traditions of dragon boat racing, eating zongzi (sticky rice dumplings), and other customs such as hanging calamus and mugwort, drinking realgar wine, and wearing perfume pouches to ward off evil spirits and disease."`
この定式化により、エンベディングモデルに必要な情報の豊かで一貫した記述が与えられ、ベクトル空間内で関連ドキュメントのエンベディングに近い密なベクトルが生成されます。

```json
{
    "query": {
        "neural": {
            "passage_embedding": {
                "query_text": "<enriched natural language query>",
                "model_id": "...",
                "k": 100
            }
        }
    }
}
```

#### Hybrid 検索プロンプト

Hybrid 検索では、プロンプトが両方の戦略を組み合わせ、エージェントに対して単一のクエリから 2 つの補完的な定式化を生成するよう指示します。

- BM25 コンポーネント用のキーワード定式化 (フラットなキーワードバッグ)。
- Neural コンポーネント用のセマンティック定式化 (自然言語の文)。

これにより、両方のメカニズムに単一のクエリ表現を強制するのではなく、各検索メカニズムが最も効果的に処理できるフォーマットでクエリを受け取ることができます。エージェントは両方の定式化を含む Hybrid クエリ構造を返します。

```json
{
    "query": {
        "hybrid": {
            "queries": [
                {
                    "multi_match": {
                        "query": "<keyword formulation>",
                        "type": "best_fields",
                        "fields": ["text_key", "title_key"],
                        "tie_breaker": 0.5
                    }
                },
                {
                    "neural": {
                        "passage_embedding": {
                            "query_text": "<semantic formulation>",
                            "model_id": "...",
                            "k": 100
                        }
                    }
                }
            ]
        }
    }
}
```

#### システムプロンプトのカスタマイズ

これらのプロンプト設計は、エージェンティック検索のコア原則を反映しています。システムプロンプトは重要なチューニングメカニズムです。`QueryPlanningTool` 設定の `query_planner_system_prompt` パラメータをカスタマイズして、特定のユースケース、ドメイン、検索戦略に最適化できます。評価では、`QueryPlanningTool` に直接渡すカスタムシステムプロンプトを持つ `flow` エージェントを使用しました。

```json
{
    "name": "Agent for agentic search",
    "type": "flow",
    "description": "Use this for agentic search",
    "tools": [
        {
            "type": "QueryPlanningTool",
            "parameters": {
                "response_filter": "$.output.message.content[0].text",
                "query_planner_system_prompt": "<retrieval-mode-specific prompt>",
                "model_id": "<model_id>"
            },
            "include_output_in_agent_response": false
        }
    ]
}
```

プロンプトチューニングは、正しい DSL 構文の生成だけでなく、エージェントが検索タスク自体についてどのように考えるかを決定します。システムプロンプトを検索モードに合わせて調整することで、エージェントのクエリ強化戦略が基盤となる検索メカニズムを補完し、対立しないようにします。テスト結果はこのチューニングメカニズムの影響を示しています。特定のドメインとデータに合わせてシステムプロンプトをカスタマイズすることで、同様の改善を達成できます。

#### 結果分析

各検索手法をそのエージェンティック版と比較して結果を分析しました。

**Lexical 検索**

標準 BM25 とエージェンティック Lexical 検索を比較することで、用語ベースの検索に対するキーワード拡張の影響を分離できます。BEIR データセットでは、エージェンティック Lexical 検索が 7 つのデータセットのうち 4 つで BM25 検索を上回りました。

- **NQ**: +40.9% (0.3098 → 0.4364)
- **NF-Corpus**: +16.3% (0.3035 → 0.3529)
- **TREC-COVID**: +5.1% (0.6142 → 0.6455)
- **SciFact**: +4% (0.6499 → 0.6761)

キーワード拡張戦略がこれらのデータセットで効果的なのは、ユーザーの自然言語表現が関連ドキュメント内の正確な用語と一致しないクエリが含まれているためです。同義語やドメイン固有の語彙でクエリを拡張することで、そのギャップを埋めます。

BRIGHT データセットでは、改善がはるかに顕著です。エージェンティック Lexical 検索は 6 つのデータセットのうち 5 つで標準 BM25 検索を上回りました。クエリの表面的な用語が関連ドキュメントに出現しないことが多いため、キーワードマッチングでは本質的に困難です。エージェントが関連する技術用語を使用してクエリを拡張する能力が、これらの推論集約型タスクの検索品質を変革します。

- **Biology**: +190% (0.08499 → 0.24645)
- **Psychology**: +174% (0.06316 → 0.17316)
- **StackOverflow**: +123% (0.12144 → 0.27023)
- **Earth Science**: +71% (0.10753 → 0.18398)
- **Economics**: +32% (0.09711 → 0.12788)

高度にドメイン固有の表現を持つデータセット (FiQA、ArguAna、SciDocs、LeetCode) では、元のクエリ用語が関連ドキュメントに対して既に正確にマッチしている場合、キーワード拡張がノイズを導入するため、エージェンティック Lexical 検索のパフォーマンスが低下します。

**Neural 検索**

標準 Neural 検索とエージェンティック Neural 検索を比較することで、エンベディングベースの検索に対するセマンティック強化の価値を示します。BEIR データセットでは、エージェンティック Neural 検索が 7 つのデータセットのうち 5 つで標準 Neural 検索を上回りました。強化された自然言語の再定式化により、エンベディングモデルにユーザーインテントのより完全な記述が与えられ、クエリの完全なセマンティックな意味をより適切に捉えるベクトルが生成されます。

- **NQ**: +32.3% (0.33252 → 0.43972)
- **TREC-COVID**: +87.8% (0.334 → 0.62718)
- **SciFact**: +26.9% (0.4665 → 0.59196)
- **SciDocs**: +14.0% (0.1317 → 0.1501)
- **NF-Corpus**: +8.5% (0.2588 → 0.2806)

BRIGHT データセットでは、結果はまちまちですが、一部のデータセットでは改善が見られます。標準 Neural 検索は、短く曖昧なクエリにより関連ドキュメントベクトルの近くに位置しないエンベディングが生成されるため、ほとんどの BRIGHT データセットでパフォーマンスが低くなります。エージェントのセマンティック強化は、これらのクエリをエンベディングモデルが効果的に使用できる豊かな記述に変換します。

- **Biology**: +1911% (0.01417 → 0.28487)
- **Psychology**: +446% (0.03715 → 0.20294)

エージェンティック Neural 検索は StackOverflow などのデータセットでパフォーマンスが低下し、他の BRIGHT データセットでも限定的な改善にとどまります。これはエージェンティック Lexical 検索で見られた問題と同様です。クエリの強化された再定式化が元のユーザーインテントからドリフトする可能性があるか、エンベディングモデル自体がそのドメインのセマンティクスを十分に捉えられない可能性があります。

**Hybrid 検索**

エージェンティック Hybrid 検索は、元のクエリテキストのデュアル再定式化の複合効果を示します。各検索手法が最適なフォーマットでクエリを受け取ります。BEIR ベンチマークでは、エージェンティック Hybrid 検索が 7 つのデータセットのうち 5 つで最高の NDCG@10 を達成しました。特に以下のデータセットで改善が顕著です。

- **NQ**: 標準 Hybrid に対して +36.6% の改善 (0.3926 → 0.5361)。最も広く使用されている検索ベンチマークの 1 つでの大幅な改善。
- **TREC-COVID**: +23.6% の改善 (0.5503 → 0.6802)。ドメイン固有の生物医学検索での強力なパフォーマンスを実証。
- **NF-Corpus**: +19.0% の改善 (0.3086 → 0.3672)
- **SciFact**: +8.6% の改善 (0.6397 → 0.6945)
- **SciDocs**: +10.7% の改善 (0.1665 → 0.1844)

FiQA と ArguAna の 2 つのデータセットでは、エージェンティック手法が標準手法と比較してわずかに低下しました。FiQA の金融意見クエリと ArguAna の反論検索タスクは、エージェントのクエリ再定式化が元のクエリインテントからドリフトする可能性がある、微妙でドメイン固有の言語を含んでいます。

推論集約型の検索をテストするために設計された BRIGHT ベンチマークでは、エージェンティック検索が 6 つのデータセットのうち 5 つで大幅な改善を達成しました。

- **Biology**: 最良の標準手法に対して +235% の改善 (0.0850 → 0.2849)
- **Psychology**: +224% の改善 (0.0728 → 0.2358)
- **StackOverflow**: +122% の改善 (0.1214 → 0.2702)
- **Earth Science**: +118% の改善 (0.1075 → 0.2342)
- **Economics**: +38% の改善 (0.0971 → 0.1342)

これらの結果は、複雑なクエリに対する検索品質の変革的な改善を表しています。BRIGHT ベンチマークは、より深い理解を必要とするクエリに対する従来の検索手法の限界を明らかにするために特別に設計されました。これはまさに、クエリの推論と再定式化が最も価値を発揮するシナリオです。

唯一の例外は LeetCode データセットで、標準 Hybrid 検索がすべてのエージェンティック検索構成をわずかに上回りました。プログラミング問題の検索は、元のクエリが直接マッチングに既に適している高度に特定の技術用語を含む可能性があり、再定式化は正確な用語を希釈するリスクがあります。

#### 主な知見

関連性評価からいくつかの明確なパターンが浮かび上がります。

**1. エージェンティック検索は複雑で推論集約型のクエリに優れる**: BRIGHT データセットの結果は、クエリが難しいほどエージェンティック検索がより多くの価値を提供することを示しています。表面的なキーワードを超えたインテントの理解が必要なクエリでは、エージェントがクエリを分析、分解、再定式化する能力が劇的に優れた検索結果につながります。

**2. Hybrid 検索はエージェンティック検索の最も強力な基盤**: BEIR と BRIGHT の両方のデータセットで、エージェンティック Hybrid 構成が最も頻繁に最高のパフォーマンスを達成しました。Lexical の精度とセマンティックな理解を組み合わせ、デュアル定式化プロンプトを通じて各メカニズムに合わせたクエリを受け取ることで、Hybrid 検索はエージェントに最も柔軟な検索基盤を提供します。

**3. 標準ベンチマークでは堅実な改善、難易度の高いベンチマークでは変革的な改善**: BEIR データセットでは、エージェンティック手法が 7 つのデータセットのうち 5 つで NDCG@10 を改善し、改善が見られたデータセットでの平均改善率は約 15〜20% でした。BRIGHT データセットでは、改善が見られたデータセットでの平均改善率は 100% を超えました。これは、エージェンティック検索の価値がクエリの複雑さとともに増加することを示唆しています。

**4. 検索モードを意識したプロンプティングが違いを生む**: キーワードバッグとセマンティック文の両方を生成するデュアル定式化プロンプトを使用するエージェンティック Hybrid 検索の強力なパフォーマンスは、システムプロンプトを検索メカニズムに合わせて調整することの有効性を検証しています。これは `query_planner_system_prompt` 設定を通じてすべてのエージェンティック検索ユーザーが利用できるメカニズムです。

**5. 一部のクエリタイプはエージェンティック再定式化の恩恵が少ない**: FiQA、ArguAna、LeetCode などのデータセットは、エージェンティック再定式化が普遍的に有益ではないことを示しています。検索に対して既に適切に構成されているクエリや、非常に正確な用語を必要とするクエリは、再定式化の恩恵を受けない場合があり、場合によっては悪影響を受けることもあります。いつ再定式化し、いつ元のクエリをそのまま渡すかを理解することは、今後の最適化の重要な方向性です。

## 実行精度

関連性が返されたドキュメントの品質と位置に関するものである一方、実行精度は生成されたクエリ自体の品質に関するもので、分析やアグリゲーションに関連するユーザーの質問を LLM が正確に変換する能力を評価しようとするものです。私たちのコアとなる問いは「エージェンティック検索は自然言語を正しい構造化クエリにどの程度正確に変換するか」でした。

この問いに答えるため、厳密な評価フレームワークが必要でした。テキストから SQL へのシステムの確立されたベンチマークである [Spider データセット](https://yale-lily.github.io/spider)を選択し、自然言語を OpenSearch クエリ DSL (ドメイン固有言語) に変換するエージェンティック検索の能力を評価するために適応させました。

### Spider データセットを選んだ理由

Spider は Yale 大学の研究者が作成した大規模なセマンティックパーシングデータセットで、138 の異なるドメインにまたがる 200 のデータベースにわたって 7,000 の質問と SQL クエリのペアが含まれています。元々はテキストから SQL へのシステム (自然言語の質問を SQL クエリに変換するモデル) のベンチマーク用に設計されました。SQL と OpenSearch のクエリ DSL はどちらも類似した操作を表現する構造化クエリ言語であるため、エージェンティック検索のテキストから DSL への変換能力の評価に最適です。以下の表に SQL と OpenSearch クエリ DSL の操作の比較を示します。

| 操作 | SQL | OpenSearch クエリ DSL |
|:---|:---|:---|
| フィールド選択 | `SELECT` | `_source` / `fields` |
| フィルタリング | `WHERE` | `query` 句 |
| アグリゲーション | `GROUP BY` | `aggs` |
| ソートと制限 | `ORDER BY` / `LIMIT` | `sort` / `size` |

"Show me all customers from California who spent more than $1000 last month" を SQL または DSL に変換する認知的な課題は基本的に同じです。インテントを理解し、関連するフィールドを特定し、適切なフィルタロジックを構築することです。

Spider の 138 の異なるドメインは、システムが馴染みのないスキーマにわたってクエリ理解を汎化できるかどうかをテストします。これは、ユーザーがさまざまなフィールド構造とデータ型を持つ多様な OpenSearch インデックスにクエリを実行する実際のエージェンティック検索の使用を表しています。クエリはシンプルなもの ("How many employees are there?") から複雑なもの ("What are the name and budget of the departments with average instructor salary greater than the overall average?") まで幅広く、ユーザーが尋ねる可能性のある質問の全範囲を処理するエージェンティック検索の能力をテストします。

### Spider データセットの OpenSearch への適応

データセットのクエリが OpenSearch にうまく変換されるよう、データセットの広範なフィルタリングを行いました。Spider データセットの SQL クエリは、データが複数の関連テーブルに分散していることを前提とする JOIN やサブクエリ (`SELECT`、`UNION`、`INTERSECT`、`EXCEPT` 句) などの機能を使用しています。一方、OpenSearch は外部キーでリンクされたテーブルではなく、自己完結型のドキュメントとしてデータを保存するため、評価は単一テーブル操作に焦点を当てました。

公正な評価を作成するため、これらのクエリをフィルタリングし、フィルタリング、ソート、アグリゲーションなどの単一テーブル操作に実験を集中させました。元の 7,000 の Spider サンプルから 921 を評価しました。

### 評価方法

以下のプロセスでエージェンティック検索を評価しました。

1. **ゴールド SQL 実行**: 人間が検証した「ゴールド」SQL クエリを SQLite テーブルに対して実行し、参照 DataFrame を生成。
2. **エージェンティッククエリ実行**: 対応する自然言語の質問を使用してエージェンティック検索クエリを実行し、結果 DataFrame を生成。
    ```
    GET <index_name>/_search?search_pipeline=agentic-search-pipeline
    {
        "query": {
            "agentic": {
                "query_text": "<Natural Language Question>"
            }
        }
    }
    ```
3. **DataFrame 比較**: 検索結果を比較して同じデータが含まれているかどうかを判定。

有効なクエリのバリエーションを考慮するために設計された以下の評価基準を使用しました。

- 行/列の順序と列名に関係なく結果を比較。
- 列名が異なる場合、列の値が同一かどうかを検証。
- 予測された DataFrame に追加の列が含まれていてもよい。すべてのゴールド列が正確に一致する行で含まれていれば正解とみなす。

このアプローチは、同じ質問に対して複数の有効な回答方法があることが多いことを認識しています。同じ SQL クエリが異なる方法で記述されても同一の結果を返すのと同様です。

### 結果

以下に実行精度の評価結果を示します。

```

============================================================
Total examples attempted: 921

agentic search:

  Correct: 604
  Execution Accuracy: 82.07%

```

エージェンティック検索は、5 回中 4 回以上 (82% の実行精度) で自然言語を構造化クエリに正しく変換します。クエリ構文やスキーマを理解しなくても、平易な英語でデータにクエリを実行できます。

#### システムプロンプトチューニング

実行精度の評価から得られた重要な知見は、関連性の評価からの発見を補強するものですが、システムプロンプトの設計が検索品質にとって重要なメカニズムであるということです。最適な実行精度を達成するには、エージェンティック検索で使用されるシステムプロンプトの慎重なチューニングが必要であることを観察しました。当初、システムプロンプトにはモデルを正しいクエリ DSL 構文に導くためのサンプルクエリが含まれていました。しかし、より新しく高性能なモデルがこれらのサンプルに過学習する傾向があることを観察しました。適切でない場合でもサンプルパターンに厳密に従ったり、サンプル構造をそのまま再現したりして、不正確なフィルタロジックにつながりました。

デフォルトの検索テンプレートとサンプルを削除することで、モデルがユーザーの自然言語の質問、インデックスマッピング情報、およびモデル固有のクエリ構築の理解に基づいて各クエリについて推論できるようにしました。このアプローチにより、多様な Spider データベースにわたってより良い汎化が得られ、モデルがクエリを事前定義されたパターンに強制するのではなく、各固有のマッピングに合わせてクエリ構築戦略を適応させることができました。

これはエージェンティックシステムにとって重要な知見を浮き彫りにしています。基盤となるモデルがより高性能になるにつれて、より規範的でないプロンプティングがより良い結果を生むことが多いということです。モデルが進化し改善されるにつれて、継続的な評価とプロンプトの反復が不可欠です。

## 検索関連性と実行精度の結果

以下の表に、検索関連性と実行精度の統合結果を示します。

| タスク | ベンチマーク | 結果 |
|:---|:---|:---|
| 関連性 (標準) | BEIR | 7 データセット中 5 つで最高の NDCG@10。最大 +36.6% の改善 |
| 関連性 (高難度) | BRIGHT | 6 データセット中 5 つで最高の NDCG@10。最大 +235% の改善 |
| 実行精度 | Spider | 自然言語の質問の 82% が構造化クエリに正しく変換 |

これらの結果は、両方の検索パラダイムにわたってエージェンティック検索を検証しています。構造化クエリ (フィルタリング、ソート、アグリゲーション) で 82% の実行精度、非構造化テキスト検索で大幅な関連性の改善を達成し、最も困難なクエリで最も劇的な改善が見られました。

これらの結果を合わせると、エージェンティック検索が OpenSearch の使用における技術的障壁を低減することが実証されます。現在、OpenSearch を効果的に使用するには、クエリ DSL 構文、クエリタイプ (`match`、`term`、`bool`)、句ロジック (`must`、`should`、`filter`)、アナライザの動作、アグリゲーション構文、関連性チューニングの知識が必要です。そのため、ビジネスアナリスト、プロダクトマネージャー、その他の非技術ユーザーは、クエリの作成をエンジニアに依頼する必要があります。エージェンティック検索はこの障壁を取り除きます。ユーザーは自然言語で質問し、正確で適切にランク付けされた結果を得ることができます。

## 今後の予定

エージェンティック検索のパフォーマンスと関連性をさらに向上させることを目的とした機能に注力しています。例えば、エージェンティックメモリ、リランキング、インデックスパターンとエイリアスを使用した検索のサポート、検索テンプレート統合の最適化などです。

フィードバック、質問、または実装してほしい追加のユースケースのアイデアがあれば、ぜひお聞かせください。[OpenSearch フォーラム](https://forum.opensearch.org/t/use-cases-and-general-feedback-for-agentic-search/27488)または [OpenSearch Slack ワークスペース](https://opensearch.org/slack/)でディスカッションに参加してください。

進行中の開発と設計の更新を追跡するには、[エージェンティック検索 RFC](https://github.com/opensearch-project/neural-search/issues/1525) を参照してください。

## 謝辞

評価フレームワークは、OpenSearch のシニアアプライドサイエンティストである Alexander Greaves-Tunnell の基礎的な研究に基づいています。彼はテキストから PPL (Piped Processing Language) モデルへの変換精度を測定するための元の評価方法論を開発しました。クエリ変換精度を測定する彼の厳密なアプローチが、エージェンティック検索の実行精度評価ツールの基盤を提供しました。このバージョンは、エージェンティック検索のテキストから DSL への変換能力をサポートするために彼の元のフレームワークを拡張したものです。
