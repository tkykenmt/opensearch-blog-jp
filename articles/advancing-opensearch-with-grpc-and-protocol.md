REST API は数十年にわたり Web サービスの基盤として機能し、シンプルさと幅広い互換性を提供してきました。しかし、データ集約型のアプリケーションがスケールするにつれて、REST API で使用されるテキストベースの JSON フォーマットは、パース処理のオーバーヘッドやペイロードサイズの増大によりボトルネックとなる可能性があります。gRPC と Protocol Buffers (Protobuf) を組み合わせた最新の代替手段は、バイナリシリアライゼーション、HTTP/2 多重化、強い型付けによってこれらの制約に対処します。

業界ではこれらのメリットが広く認識されており、Google、Netflix、Uber などの企業が社内マイクロサービスに gRPC を採用しています。OpenSearch もこの流れに加わり、Protobuf スキーマを使用するネイティブ gRPC エンドポイントを導入することで、従来の REST API に代わる高パフォーマンスな選択肢を提供します。

本記事では、一貫性のある REST と gRPC の API を維持する際の課題、OpenSearch の OpenAPI 仕様から Protobuf 定義を自動生成するアプローチ、そして実際のパフォーマンス向上を示すベンチマーク結果について解説します。

## Protobuf の生成と自動化

既存の REST API フレームワークに gRPC サポートを追加するには、いくつかの課題があります。

1. **一貫性の維持**: REST と gRPC の API は同一の機能を公開し、同じ入力を受け付け、同等の出力を返す必要があります。手動での同期はエラーが発生しやすく、スケールしません。
2. **複雑なセマンティクスの表現**: OpenSearch の REST API には、ポリモーフィックなリクエストボディ、条件付きフィールドバリデーション、動的マッピングなど、複雑な動作があり、これらを Protobuf スキーマで正確に表現する必要があります。
3. **API の同期維持**: OpenSearch の進化に伴い、REST と gRPC の両方の API を同時に更新する必要があります。手動でのメンテナンスはドリフトや技術的負債を生み出します。

これらの課題に対処するため、既存の OpenSearch OpenAPI 仕様から Protobuf スキーマを直接生成する自動変換パイプラインを開発しました。これにより、REST と gRPC の API がソースレベルで同期された状態を維持できます。

### 変換パイプラインの概要

変換パイプラインは、以下の図に示すように、前処理、カスタマイズされた OpenAPI 生成、後処理の 3 つのステージで構成されています。

![変換パイプライン](/images/advancing-opensearch-with-grpc-and-protocol/4f146234476c.png)

以下のセクションでは、各ステージについて説明します。

### ステージ 1: 前処理

パイプラインはまず OpenAPI 仕様を正規化します。

- すべての `$ref` ポインタをインライン定義に解決します。
- ネストされたスキーマをフラット化し、よりクリーンな Protobuf メッセージ生成を実現します。
- 必須フィールドと型制約を検証します。
- 命名規則を Protobuf 互換の形式に標準化します。

### ステージ 2: コア変換

OpenAPI Generator ツールを使用して、前処理された仕様を Protobuf に変換します。

- OpenAPI オブジェクトは Protobuf の `message` 型になります。
- REST エンドポイントは `rpc` メソッドを持つ gRPC の `service` 定義にマッピングされます。
- JSON のプリミティブ型は Protobuf のスカラー型に変換されます。
- 配列は `repeated` フィールドに変換されます。

このツールは OpenSearch の API パターンに固有のカスタムテンプレートと設定を適用し、ポリモーフィック型やオプショナルフィールドなどのエッジケースを処理します。

### ステージ 3: 後処理と互換性チェック

最終ステージでは、生成された Protobuf スキーマを検証します。

- **ワイヤ互換性の検証**: REST の JSON と gRPC の Protobuf 表現が同等の構造にシリアライズおよびデシリアライズされることを確認します。
- **セマンティック検証**: フィールド制約、デフォルト値、バリデーションルールがプロトコル間で一致することを確認します。
- **回帰テスト**: 生成されたスキーマを以前のバージョンと比較し、破壊的変更を防止します。
- **カスタム調整**: 自動変換では表現しきれない微妙な API の動作に対して、手動での調整を適用します。

このパイプラインは CI システムの一部として実行されます。開発者が OpenAPI 仕様を変更すると、パイプラインが自動的に Protobuf スキーマを再生成し、互換性を検証するため、2 つの API が乖離することはありません。

## OpenSearch のネイティブ gRPC サポート

OpenSearch の gRPC 実装は、プロキシやアダプタレイヤーとしてではなく、OpenSearch ノード内で直接動作します。gRPC トランスポートは REST トランスポートと並行して動作し、両者は同じリクエストハンドラとビジネスロジックを共有します。

このアプローチには以下のメリットがあります。

- **追加のインフラストラクチャが不要**: 別途 gRPC ゲートウェイサービスをデプロイする必要がありません。
- **一貫した動作**: 両方のプロトコルが同一のコードパスを実行するため、機能的な同等性が保証されます。
- **段階的な導入**: 一部のエンドポイントで gRPC を使用しながら、他のエンドポイントでは REST を維持するという段階的な導入が可能です。両プロトコルは異なるポートで動作するため、同時に有効化できます。

## ベンチマーク実験: gRPC と REST の比較

gRPC のパフォーマンスへの影響を定量化するため、ベクトル検索とバルクインジェストという 2 つの一般的なワークロードで gRPC と REST を比較するベンチマークを実施しました。

### テスト構成

以下のテスト構成を使用しました。

- **クラスタ構成**:
  - OpenSearch バージョン 3.3
  - 3 台の `c5.xlarge` クラスタマネージャーノード
  - 5 台の `r5.xlarge` データノード
- **ベンチマークツール**: OpenSearch Benchmark (OSB)
- **ネットワーク**: ネットワークの変動を最小化するため、すべてのノードを同一の AWS アベイラビリティゾーンに配置

### k-NN ベクトル検索

セマンティック検索やレコメンデーションシステムで一般的な、レイテンシに敏感なワークロードである k 近傍法 (k-NN) ベクトル検索のベンチマークを実施しました。このワークロードのデータセットは、[`mxbai-large-v1`](https://huggingface.co/mixedbread-ai/mxbai-embed-large-v1) を使用した MS MARCO のランダムサンプル (インジェスト済みの 100 万ベクトルに対する 10 万クエリ、合計 4.3 GB) です。

以下の表にレイテンシの比較を示します。

| パーセンタイル | REST (ms) | gRPC (ms) | 改善率 |
|:---:|:---:|:---:|:---:|
| P50 | 5.33 | 5.08 | 4.74% |
| P90 | 6.00 | 5.64 | 5.98% |

以下の表にスループットの比較を示します。

| REST (ops/sec) | gRPC (ops/sec) | 改善率 |
|:---:|:---:|:---:|
| 143.26 | 173.12 | 17.24% |

テストでは、gRPC は一貫して **4.74%** の P50 レイテンシ改善を達成しました。平均スループットは **17.24%** 向上し、同じクラスタでより多くのクエリを毎秒処理できるようになりました。これらの改善は以下の要因によるものです。

- Protobuf のコンパクトなバイナリエンコーディングによる**ペイロードサイズの 53% 削減**。
- シリアライゼーションとデシリアライゼーションにおける**クライアント側の処理時間の約 58% 削減**。
- クライアント側とサーバー側の両方での CPU 使用率の低下。

### バルクインジェスト

`opensearch-benchmark-workloads` リポジトリの `http_logs` データセット (2 億 4,700 万ドキュメント、合計約 31 GB) を使用して、バルクドキュメントインデキシングをテストしました。

以下の表にバルクリクエストサイズ別のパフォーマンスを示します。

| リクエストあたりのドキュメント数 | gRPC スループット改善率 | gRPC レイテンシ削減率 (P50) |
|:---:|:---:|:---:|
| 10,000 | 16.2% | 22.6% |
| 5,000 | 15.8% | 22.4% |
| 2,500 | 14.8% | 21.1% |
| 1,000 | 12% | 13.6% |

主な発見は以下のとおりです。

- 異なるバルクサイズにわたる一貫した約 **22% のレイテンシ削減**。
- バイナリエンコーディングによる平均 **16.4% のペイロードサイズ削減**。
- パフォーマンス改善はバルクリクエストあたり約 5,000 ドキュメントで頭打ちとなり、このデータセットにおける最適なバッチサイズを示唆しています。

バルクインジェストが gRPC から大きな恩恵を受ける理由は以下のとおりです。

- 大きな JSON 配列は圧縮効率が低く、高コストなパース処理が必要です。
- HTTP/2 多重化により、単一の接続上で同時にバルクリクエストを送信できます。
- バイナリエンコーディングによりネットワーク転送時間が短縮されます。これは特に高スループットのインジェストパイプラインで重要です。

### バイナリドキュメントフォーマット

API 構造の Protobuf に加えて、OpenSearch はインデックスされる実際のコンテンツに対して**バイナリドキュメントフォーマット**をサポートしています。

- **SMILE**: Jackson プロジェクトが管理する JSON のバイナリエンコーディング。
- **CBOR (Concise Binary Object Representation)**: JSON に類似した IETF 標準のバイナリフォーマット。

これらのフォーマットを gRPC と組み合わせることで、ペイロードサイズとパースのオーバーヘッドをさらに削減できます。例えば、SMILE と gRPC を組み合わせると、REST と JSON の使用と比較して最大 **65% のペイロード削減**を達成できます。

バイナリフォーマットは `Content-Type` ヘッダーで指定でき、OpenSearch がシリアライゼーションとデシリアライゼーションを自動的に処理します。実装の詳細については、[issue #19311](https://github.com/opensearch-project/OpenSearch/issues/19311) を参照してください。

## ベストプラクティス

ベンチマークと本番環境での経験に基づき、gRPC は以下のシナリオで最も効果を発揮します。

1. **大きなペイロード**: リクエストまたはレスポンスのサイズが数キロバイトを超える場合、Protobuf のバイナリエンコーディングにより測定可能なパフォーマンス向上が得られます。
2. **高スループットワークロード**: 毎秒数千のオペレーションを必要とするアプリケーションは、gRPC の低い CPU オーバーヘッドと HTTP/2 の接続効率の恩恵を受けます。
3. **バイナリドキュメントフォーマット**: gRPC と SMILE または CBOR を組み合わせることで、ペイロード削減とパース速度を最大化できます。

これらの条件が当てはまらないユースケース (低頻度の管理操作やインタラクティブなデバッグなど) では、REST がよりシンプルでアクセスしやすい選択肢です。

gRPC が移行を正当化するのに十分なメリットを提供するかどうかを判断するために、特定のワークロードとデータ特性を使用したエンドツーエンドのベンチマークを推奨します。

## 現在の状況と今後の予定

OpenSearch の gRPC サポートは、以下を含む一部の API で利用可能です。

- Bulk Indexing API
- k-NN ベクトル検索
- Search API (実験的)

OpenSearch Benchmark (OSB) はこの実装をサポートしており、パフォーマンステスト用のネイティブ gRPC ワークロード設定が含まれています。

### 今後の展開

gRPC サポートを OpenSearch 全体に積極的に拡大しています。

1. **より広い API カバレッジ**: アグリゲーション、スナップショット、クラスタ管理、その他の頻繁に使用される API に gRPC エンドポイントを追加します。
2. **クライアントライブラリのサポート**: 導入を簡素化するため、Java、Python、JavaScript、Go 向けの公式 gRPC クライアントを開発します。
3. **セキュリティとオブザーバビリティ**: 認証と認可のために gRPC を OpenSearch Security プラグインと統合し、gRPC リクエストのロギングとトレーシングを強化します。
4. **パフォーマンスの最適化**: ストリーミング検索結果、サーバーサイドバッチング、カスタム圧縮アルゴリズムなどの高度な技術を検討します。

gRPC は REST を置き換えるものではなく、パフォーマンスが重要なシナリオで優れた補完的なオプションです。OpenSearch は両方のプロトコルを引き続きサポートし、ユースケースに最適なものを選択できるようにします。
