OpenSearch が進化を続ける中、明確さとコラボレーションがこれまで以上に重要になっています。私たちは OpenSearch の技術ロードマップの共有方法を変革し、次に何が来るのかをより見通しやすくしています。この新しいアプローチにより、フィードバックの提供、機能の優先順位付け、開発へのコラボレーションが容易になり、コントリビューターの取り組みがコミュニティのニーズと一致するようになります。プロジェクトは、4 つのイノベーションの柱に基づいたテーマベースのコミュニティ主導ロードマップを導入しました。

- 検索のモダナイゼーション
- オブザーバビリティと分析
- スケーラビリティとレジリエンシー
- コミュニティとプラットフォーム

これらの柱は、OpenSearch を検索、分析、オブザーバビリティのための推奨されるオープンソースソリューションとして、現在も将来も位置づけ続ける明確な開発フレームワークを構築します。

本記事では各柱を探り、まず概要を説明してから技術的な詳細に踏み込みます。記事全体を通じて、関連する GitHub RFC や Issue へのリンクがあり、具体的なイノベーションの探索、フィードバックの提供、開発への貢献が可能です。

## 2026 年の概要

OpenSearch は、あらゆる規模の組織向けに AI システムを動かすために構築された、一貫性のある AI ネイティブプラットフォームへと進化しています。プロジェクトは、ハイブリッド検索、ベクトル検索、セマンティック検索、クロスエンコーダー、リランキングなどの高度な技術を、Python ネイティブの開発者体験を備えたコンポーザブルなクエリパイプラインに簡素化することで、クエリの複雑さのギャップを埋めています。OpenSearch は、ビジネスコンテキストを捉えるクエリ理解、ネイティブの条件ロジック、インターリーブ A/B テストなど、洗練された機能をすぐに使える形で提供します。

コアエンジンのモダナイゼーションは、2 倍のスループット改善、ストリーミングアグリゲーション、gRPC API により、AI エージェントの高スループット要求をサポートします。Model Context Protocol (MCP) 統合と AgentHealth 評価ツールによるネイティブのエージェントサポートは、AI エンジニアが RAG とエージェントソリューションを大規模に構築するのを支援します。

検索のモダナイゼーションは、組み込みのクエリ理解とインターリーブ A/B テストによる評価で「すぐに動く」機能を提供し、OpenSearch が AI エージェントを新しいクラスの検索ユーザーとしてサポートするように進化する中で、AI ネイティブワークロードの要求に応えるためにベクトルと検索の機能を向上させます。

オブザーバビリティと分析は、ログ、メトリクス、トレースを統合する統一プラットフォームを構築します。Discover を分析の中心ハブとして、OpenSearch はワークフローを合理化し、ユーザーがコンテキストスイッチなしにクエリから可視化、アラートへと移行できるようにします。主な取り組みには、OpenTelemetry (OTLP) サポートの深化、PPL と SQL インターフェースの強化、OpenSearch のオブザーバビリティツールキットと Prometheus の統合による統一オブザーバビリティの実現が含まれます。

スケーラビリティとレジリエンシーは、成長段階ごとにクラスター全体の再設計を必要としないシームレスなスケーリングを提供します。これには、コストとパフォーマンスのバランスを取るためのリモートバックドストレージと自動ストレージティアリングの推進、および暴走クエリがクラスターの安定性に影響を与えるのを防ぐ検索バックプレッシャーなどのクエリレジリエンシー機能の実装が含まれます。ビジョンは、自動化されたゼロコンフィグレーションスケーリングにより、運用のシンプルさとペタバイトスケールまでのコスト効率の良い成長を実現するクラウドネイティブアーキテクチャです。

コミュニティとプラットフォームは、OpenSearch Project の健全性と成長に取り組みます。プロジェクトは、コントリビューターの明確な昇進パスを定義し、アクティブな参加を増やすことで、メンテナーシップの拡大にコミットしています。これには、迅速で自動化されたリリースプロセスの維持と、機能の整合性を保護するための透明なメトリクスの提供が含まれます。プロジェクトは、エージェント検索システムの提供、Python ネイティブ環境との統合の容易化、コントリビューターがパブリックロードマップのアイデアを提案・分類する方法の簡素化により、新しいユーザーコミュニティをサポートします。

## 4 つの柱: 詳細

以下のセクションでは、各柱がどのように実用的な機能とイノベーションに変換されるかを説明します。OpenSearch コミュニティの一員として、これらを採用、評価、形作ることができます。

### 柱 1: 検索のモダナイゼーション

**AI ネイティブ検索プラットフォーム**は、エンタープライズ AI システムの統一バックボーンとして機能します。単純なクエリに対してデータを返すだけでなく、OpenSearch は会話型エージェント、セキュリティ自動化、大規模なナレッジディスカバリーをサポートします。プロジェクトの計画されたイノベーションは、複雑なレガシー設定や冗長な JSON から離れ、AI エンジニアやプロダクトマネージャーに直接訴求する体験を提供します。これらの高度な機能を統合することで、OpenSearch は開発から本番まで AI 検索のフルライフサイクルを処理します。

**AI ネイティブの開発者体験**は、情報検索に不慣れなデータサイエンティストやエンジニアに、最新の機械学習ツールに見られる使いやすさを反映した Python ネイティブ環境を提供します。SearchBuilder により、Python ネイティブの AI エンジニアは、ラップトップ上のシンプルなチャットベースの UI を通じて、洗練されたクエリ構造を持つ完全な OpenSearch インデックスをすぐにプロトタイプできます。ローカル設定はそのままクラウドにデプロイできます。コンポーザブルなクエリパイプラインにより、ユーザーは前処理、ハイブリッド検索、リランキングコンポーネントを広範な手動コーディングなしにスタックできます。インターリーブ A/B テストを含む実験のネイティブサポートにより、開発者は組み込みのフィードバックループを使用してモデルを改良できます。

**洗練されたクエリ機能**は、検索拡張生成 (RAG) とエージェントソリューションを動かすための高度な機能を即座に提供します。OpenSearch は、ハイブリッド検索、クロスエンコーダー、高度なランキング関数のための簡素化されたすぐに使えるサポートを提供することで、クエリの複雑さのギャップを埋めます。ロードマップには、キーワードマッチングを超えて真のビジネスコンテキストを捉える強化されたクエリ理解が含まれます。例えば、カジュアルにブラウジングしているユーザーと購入準備ができているユーザーを区別するなどです。プロジェクトは、複雑な検索ワークフローを内部的に処理するために、クエリパイプライン内にネイティブの条件ロジックを実装する実験を行っています。

**コアエンジンのモダナイゼーション**は、AI エージェントを主要なデータ取得インターフェースとして有効にし、2026 年末までに 3 つの機能を通じて 2 倍のスループット改善を目標としています。

- **ストリーミングアグリゲーション**はミリ秒単位で段階的な結果を提供し、体感レイテンシーを 3 倍改善し、リソース使用率を 2 倍削減します。
- **エージェントネイティブの gRPC API** は中断可能なマルチターン会話をサポートし、JSON シリアライゼーションを Protobuf に置き換えることで最大 2 倍のパフォーマンス向上を実現します。
- **マルチエンジン、マルチフォーマットクエリ実行**は、検索インデックスと Parquet や Iceberg などのオープンフォーマット間でクエリをルーティングします。

これらの機能は、ベクトル化実行、ストリーミングオペレーター、プラガブル実行エンジン、ゼロコピーデータアクセスによってサポートされます。

**エージェントシステムのネイティブサポート**は、メモリコンテナ、ツールルーティング、コンテキストウィンドウの制限やハルシネーションなしに数百のツールを管理する MCP を含む、組み込みのエージェントコンテキストエンジニアリングスタックを提供します。[AgentHealth](https://github.com/opensearch-project/agent-health) は、エージェントが問題を解決する成功度を評価します。これらの機能により、Agentic Relevance Tuning (ART) が実現されます。これは、複雑な関連性の課題を安全かつ効率的に解決するエージェントプロダクトであり、検索関連性を改善するためのマルチエージェントシステムを実証します。

**ベクトル検索のスケール**は、AI/ML アプリケーションを大規模に実行するために重要な価格性能の最適化に取り組みます。2026 年、OpenSearch はパフォーマンス、コスト削減、関連性の 3 つの主要分野に注力します。

**パフォーマンスの改善**には、ハードウェアアクセラレーション、バルクベクトルスコアリング、ディスク IOPS を削減するベクトルリオーダリング、Better Binary Quantization (BBQ)、グラフ品質の改善、最大 3 倍のスループット向上を可能にするセマンティックベースのデータ組織化によるスマートルーティングが含まれます。

**コスト削減**は、32 倍を超える圧縮率を実現する高度な量子化技術、重複するフラットベクトルストレージの排除、ベクトルノードのクランピングとクラスタリングによるグラフ構造の最適化によって達成されます。

**関連性の強化**には、グラフ構築にフル精度ベクトルを使用するコアアルゴリズム、ベクトルインデックスの検索プランナー、モデルベースのベクトル類似性サポートが含まれます。

### 柱 2: オブザーバビリティと分析

**ネイティブ時系列データベース (TSDB) サポート**は、大規模なメトリクスを効率的に保存するという歴史的な課題に取り組みます。プロジェクトは、時系列データ専用に設計された[ネイティブストレージエンジン](https://github.com/opensearch-project/OpenSearch/issues/19461)を導入しています。この新しいエンジンは、標準的なインデキシングの制限を超えて、テレメトリのための高性能でコスト効率の良いストレージを提供します。メトリクスの内部処理方法を最適化することで、OpenSearch は汎用データストアで通常見られるパフォーマンスの低下なしに、統一オブザーバビリティの高スケールバックボーンとして機能できます。

**Data Prepper による合理化されたインジェスト**は、新しい [TSDB](https://github.com/opensearch-project/time-series-db) エンジンへのデータの入力方法を簡素化します。プロジェクトは、Prometheus Remote Write v1 プロトコルの[サポートを追加](https://github.com/opensearch-project/data-prepper/issues/6533)することで Data Prepper を大幅に強化し、ユーザーが既存の監視パイプラインに OpenSearch をゼロフリクションで統合できるようにします。さらに、Data Prepper は OpenTelemetry (OTel) メトリクスをネイティブに TSDB フォーマットに変換し、ユーザーが OTLP 標準を採用する際にデータを自動的に最適化します。プロジェクトはまた、TSDB などの非 Prometheus ソースへの PromQL サポートの拡張を検討しており、ユーザーが既存の Prometheus の専門知識を OpenSearch 全体に適用できるようにします。

**統一された分析と開発者体験**は、異なるデータタイプ間でより直感的なフローベースのクエリ方法を提供するために、Piped Processing Language (PPL) への[投資](https://github.com/orgs/opensearch-project/projects/261/views/1)を継続します。チームがより早く開始できるように、プロジェクトは単一のコマンドラインコマンドまたは Docker Compose ダウンロードで起動するスリム化された[オブザーバビリティスタック](https://github.com/opensearch-project/observability-stack)を導入しました。これらの取り組みにより複雑さが排除され、エンジニアは数日ではなく数分でインストールからアクション可能なインサイトへと移行できます。プロジェクトはまた、Uber のメトリクス統合などの大規模な業界ユースケースを統合し、プラットフォームが世界で最も要求の厳しいオブザーバビリティワークロードに対して十分に堅牢であることを保証しています。

### 柱 3: スケーラビリティとレジリエンシー

**シームレスでインテリジェントなスケーリング**は、従来成長段階ごとにクラスター全体の再設計を必要としていたスケーリングの課題を排除します。プロジェクトは、開発者のラップトップが最大のクラウドネイティブデプロイクラスターにスケールされた場合でも同一であることを保証する、一貫したデスクトップからクラウドへのアーキテクチャを構築しています。運用オーバーヘッドを削減するために、プロジェクトはインデックスティアリングとスナップショット管理の手動スクリプトをインテリジェントな自動化に置き換え、リアルタイムのクラスター負荷に基づいて特定のコンポーネントをスキップまたは使用するリソースアウェアオプティマイザーを導入しています。

**ネイティブの Pull ベースインデキシング**は、特に予測不可能なトラフィックの急増時における従来のプッシュベースモデルの固有の制限に対処します。プロジェクトは、元々 [Uber によって開発・コントリビュート](https://www.uber.com/en-AU/blog/how-uber-indexes-streaming-data-with-pull-based-ingestion-in-opensearch/)された Pull ベースのインジェストフレームワークを推進しています。このフレームワークは、エンジンが受信プッシュリクエストに圧倒されるのではなく、外部イベントストリームから自分のペースでデータを取得できるようにすることで、クラスターの安定性を保護するネイティブのバックプレッシャー処理を導入します。このアーキテクチャの転換により、スパイク時のサーバー過負荷が防止され、インデキシングノードでの translog の必要性がなくなり、リソースオーバーヘッドが大幅に削減され、全体的な効率が向上します。

**Query Insights** は、監視ツールからクエリパフォーマンス管理プラットフォームへと進化しました。[ルールベースの推奨エンジン](https://github.com/opensearch-project/query-insights/issues/532)がクエリのアンチパターンを自動的に検出し、信頼度スコア付きのアクション可能な修正提案を生成して、Query Insights ダッシュボードに直接表示します。新しい[クエリプロファイラー UI](https://github.com/opensearch-project/query-insights-dashboards/issues/104) は、詳細な実行フェーズの検査と類似する遅いクエリとの履歴相関を提供します。トップ N クエリページには、P90/P99 メトリクス、ノード/インデックス/ユーザー/WLM グループ別のブレークダウンチャート、パフォーマンスタイムラインなどの[リッチな可視化](https://github.com/opensearch-project/query-insights-dashboards/issues/446)が含まれます。[クエリイベントモニタリング](https://github.com/opensearch-project/query-insights/issues/529)は Alerting プラグインと統合し、パフォーマンスの低下、リソーススパイク、トップ N リストに追加された新しいクエリに対するプロアクティブな通知を提供します。マルチテナントデプロイメントでは、[RBAC ベースのフィルタリング](https://github.com/opensearch-project/query-insights/issues/520)がユーザー名またはバックエンドロールによって Query Insights の可視性をスコープします。[最近完了したクエリのキャッシュレイヤー](https://github.com/opensearch-project/query-insights/issues/454)により短命なクエリが確実にキャプチャされ、[リモートリポジトリエクスポーター](https://github.com/opensearch-project/query-insights/issues/545)によりトップ N データを S3、Azure、GCS にエクスポートしてクロスクラスター分析と長期保持が可能になります。

**ワークロード管理**には、管理者がワークロードグループごとにタイムアウト、キャンセル間隔、最大同時シャードリクエスト数、アグリゲーションバケット制限のオーバーライドを定義できる[グループレベルの検索設定](https://github.com/opensearch-project/OpenSearch/issues/20555)が含まれます。これらの設定はグループに割り当てられたすべてのリクエストに自動的に適用され、クライアントがすべてのクエリでパラメータを渡す必要なく、真のマルチテナントガバナンスを実現します。[初期実装](https://github.com/opensearch-project/OpenSearch/pull/20536)は、最初の設定として phase_took から始まる基盤インフラストラクチャを提供し、追加の設定が続きます。

**高度なベクトルと ML 機能**は、AMD GPU サポートを含むより良いベクトルサポートへの投資を継続し、オンプレミスデプロイメントを可能にします。これは、独自の推論サービスで qwen3 モデルを実行する方法の簡素化など、より深い推論機能にまで及びます。プロジェクトは、リモート ML モデルを使用したインジェストパイプラインの実行のボトルネックを排除し、ユーザーが一度により多くのドキュメントを処理できるようにすることを目指しています。ロードマップには、GGUF モデルのサポートやラストトークンプーリングなどの新しい技術の探索が含まれます。

### 柱 4: コミュニティとプラットフォーム

**コントリビューターパスの改善**は、OpenSearch コントリビューターから OpenSearch メンテナーへの移行のための、より明確に[定義されたラダー](https://github.com/opensearch-project/.github/issues/195)を開発します。この取り組みは、昇進プロセスの曖昧さを排除し、重要なことに、メンテナーシップ内の組織的多様性を高めるために設計されています。リーダーシップへの透明なロードマップを提供することで、プロジェクトの将来がグローバルコミュニティ全体からの幅広い代表的な専門家グループによって形作られることを保証します。

**技術ガバナンスの進化**は、プロジェクトの成長に合わせてスケールします。プロジェクトは、Technical Steering Committee (TSC) をスケールし、増加するコントリビューション速度に対応するために全体的な技術ガバナンスを改善しています。[Build](https://github.com/opensearch-project/technical-steering/tree/main/technical-advisory-groups/build-tag) Technical Advisory Group (TAG) の成功に続き、プロジェクトは [Observability](https://github.com/opensearch-project/technical-steering/tree/main/technical-advisory-groups/observability-tag)、[Search](https://github.com/opensearch-project/technical-steering/tree/main/technical-advisory-groups/search-tag)、[Security](https://github.com/opensearch-project/technical-steering/tree/main/technical-advisory-groups/security-tag) の 3 つの新しい [TAG](https://github.com/opensearch-project/technical-steering/tree/main/technical-advisory-groups) を正式に設立しました。これらのグループは専門的な監督を提供し、これらの重要な分野でのイノベーションが一貫性を保ちコミュニティ主導であることを保証します。

**コミュニティアドボケイトの支援**は、[OpenSearch Software Foundation の Ambassador Program](https://opensearch.org/ambassadors/) を通じて、OpenSearch プラットフォームの成長に積極的に貢献するコミュニティリーダーを認定します。Ambassador は、専門知識の共有、コミュニティメンバーのサポート、世界中での OpenSearch の代表を通じて、他の人々の学習、創造、コラボレーションを支援します。彼らは知識を共有し、ユーザーグループやカンファレンスを組織・参加し、コミュニティのフィードバックをプロジェクトとファウンデーションに還元する信頼できる人々です。

**OpenSearch Software Foundation の成長**は、長期的な独立性と安定性を保証します。Executive Director の Bianca Lewis のリーダーシップの下、[ファウンデーション](https://opensearch.org/foundation/)は個人のコントリビューターと組織の両方をサポートするプログラムとイニシアチブを構築し、より深いエンゲージメントと持続可能性への道筋を作っています。

**プロフェッショナルな成長への投資**は、正式な認定プログラムを通じてユーザーのキャリア開発をサポートします。このプログラムにより、個人は OpenSearch の専門知識を検証・実証でき、検索と分析の熟練度に対する認定された基準を提供します。これらのスキルを専門化することで、プロジェクトは個人のキャリアアップを支援すると同時に、組織が AI ネイティブ検索インフラストラクチャの管理に頼ることができる認定人材の堅牢なプールを構築します。

### 参加方法

OpenSearch の強みは常にそのコミュニティにあります。最初のプルリクエストを提出する個人のコントリビューターから、本番グレードの統合を構築するエンタープライズパートナーまで、プロジェクトはオープンソースの力を信じる人々のおかげで繁栄し続けています。

ベテランのコントリビューターであっても、OpenSearch を初めて知った方であっても、このコミュニティにはあなたの居場所があります。参加する方法をいくつか紹介します。

- **コードやドキュメントのコントリビュート**: [GitHub リポジトリ](https://github.com/opensearch-project)とコントリビューションガイドラインを確認して、インパクトを与え始めましょう。
- **会話に参加**: [コミュニティフォーラム](https://forum.opensearch.org/)、[Slack](https://opensearch.org/slack/) ワークスペース、メーリングリストに参加して、質問、アイデアの共有、他のユーザーとのつながりを持ちましょう。
- **ユーザーグループへの参加、発表、リード**: OpenSearch コミュニティをあなたの街に持ち込むか、既存のグループに参加して学び、知識を共有し、ネットワークを広げましょう。
- **Issue のトリアージを支援**: [バーチャルミーティング](https://zoom-lfx.platform.linuxfoundation.org/meetings/opensearch-project?view=week)に参加して、メンテナーと一緒にオープンな Issue のトリアージに取り組みましょう。
- **コンテンツの作成**: OpenSearch についてのブログ記事を書きましょう。[ブログガイドライン](https://opensearch.org/blog/guidelines/)を確認して始めましょう。
- **ニュースレターの購読**: プロジェクトの最新情報を受信トレイで[確認](https://opensearch.org/#:~:text=Subscribe%20for%20updates%2C%20event%20info%2C%20and%20the%20latest%20community%20news)しましょう。
- **フィードバックの提供**: あなたの意見が[ロードマップ](https://github.com/orgs/opensearch-project/projects/206/views/11)を直接形作ります。GitHub Issue やコミュニティチャンネルでユースケース、機能リクエスト、提案を共有してください。

OpenSearch はコミュニティによって、コミュニティのために構築されています。プロジェクトは大小を問わずすべてのコントリビューションに感謝し、2026 年以降も検索と分析の未来を一緒に構築することを楽しみにしています。
